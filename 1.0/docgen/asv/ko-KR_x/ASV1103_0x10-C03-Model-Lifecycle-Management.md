# C3 모델 수명 주기 관리 및 변경 제어

## 제어 목표

AI 시스템은 무단 또는 안전하지 않은 모델 수정을 프로덕션에 반영하지 못하도록 변경 관리 프로세스를 구현해야 합니다. 이 제어는 개발에서 배포, 폐기까지 전체 수명 주기 동안 모델 무결성을 보장하며, 이를 통해 신속한 사고 대응이 가능하고 모든 변경 사항에 대한 책임을 유지할 수 있습니다.

핵심 보안 목표: 무결성, 추적성 및 복구 가능성을 유지하는 통제된 프로세스를 사용하여 승인되고 검증된 모델만이 생산에 도달하도록 합니다.

---

## C3.1 모델 승인 및 무결성

무결성이 검증된 승인된 모델만 생산 환경에 도달합니다.

|   #   | 설명                                                                                                                                                                             | 레벨  | 역할  |
| :---: | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | :-: | :-: |
| 3.1.1 | 모든 모델 아티팩트(가중치, 구성, 토크나이저, 기본 모델, 미세 조정, LoRA와 같은 어댑터, 안전/정책 모델 포함)가 권한 있는 엔터티에 의해 암호화되어 서명되었으며, 배포 승인 시점(및 로드 시)에 검증되어 서명되지 않았거나 변조된 아티팩트는 차단되는지 확인하십시오.                      |  1  | D/V |
| 3.1.2 | 의존성 추적이 모델 레지스트리 및 계보/의존성 그래프를 통해 실시간 재고를 유지하고, 각 환경(예: 개발, 스테이징, 프로덕션, 지역)별로 모든 소비 서비스/에이전트를 식별할 수 있는 기계 판독 가능한 모델/AI 자재 명세서(MBOM/AIBOM)(예: SPDX 또는 CycloneDX)를 생성하는지 검증하십시오. |  2  |  V  |
| 3.1.3 | 모델 출처 무결성 및 추적 기록에 권한 부여 기관의 신원, 학습 데이터 체크섬, 합격/불합격 상태가 포함된 검증 테스트 결과, 서명 지문/인증서 체인 ID, 생성 타임스탬프, 승인된 배포 환경이 포함되어 있는지 확인하십시오.                                                  |  3  | D/V |

## C3.2 모델 검증 및 테스트

모델은 배포 전에 정의된 보안 및 안전성 검증을 통과해야 합니다.

|   #   | 설명                                                                                                                                                                           | 레벨  | 역할  |
| :---: | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 3.2.1 | 모델이 배포 전에 입력 검증, 출력 정화 및 사전 합의된 조직 통과/실패 기준을 포함하는 자동화된 보안 테스트를 받도록 확인하며, 이는 에이전트 워크플로우(계획, 도구 또는 MCP 호출, RAG/메모리, 멀티모달) 및 가드레일(정책/안전 모델 또는 탐지 서비스)을 버전 관리된 평가 하네스를 통해 포괄합니다. |  1  | D/V |
| 3.2.2 | 모든 모델 변경 사항(배포, 구성, 폐기)이 타임스탬프, 인증된 행위자 신원, 변경 유형, 변경 전/후 상태, 추적 메타데이터(환경 및 사용하는 서비스/에이전트) 그리고 모델 식별자(버전/다이제스트/서명)를 포함하는 불변의 감사 기록을 생성하는지 확인하십시오.                            |  1  |  V  |
| 3.2.3 | 유효성 검사 실패가 사전 지정된 승인된 인원이 문서화된 비즈니스 정당성을 가진 명시적 승인 없이는 모델 배포를 자동으로 차단하는지 확인하십시오.                                                                                             |  2  | D/V |

## C3.3 통제된 배포 및 롤백

모델 배포는 제어되고, 모니터링되며, 되돌릴 수 있어야 합니다.

|   #   | 설명                                                                                                                          | 레벨  | 역할  |
| :---: | --------------------------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 3.3.1 | 배포 프로세스가 모델 활성화 또는 로드 전에 암호화 서명을 검증하고 무결성 체크섬을 계산하여, 불일치가 있을 경우 배포를 실패하도록 하는지 확인하십시오.                                       |  1  | D/V |
| 3.3.2 | 프로덕션 배포가 사전에 합의된 오류율, 지연 시간 임계값, 가드레일/탈옥 알림 또는 도구/MCP 실패율을 기반으로 자동 롤백 트리거가 있는 점진적 롤아웃 메커니즘(카나리 배포, 블루-그린 배포)을 구현하는지 확인하십시오. |  1  |  D  |
| 3.3.3 | 롤백 기능이 모델 상태 전체(가중치, 구성, 어댑터 및 안전/정책 모델을 포함한 종속성)를 원자적으로 복원하는지 확인하십시오.                                                      |  2  | D/V |
| 3.3.4 | 비상 모델 종료 기능이 모델 엔드포인트를 비활성화하고 에이전트 도구 또는 MCP 접근, RAG/커넥터 및 데이터베이스/API 자격 증명, 메모리 저장소 바인딩을 정의된 응답 시간 내에 비활성화할 수 있는지 확인하십시오.  |  3  | D/V |

## C3.4 보안 개발 관행

모델 개발 및 훈련 과정은 손상을 방지하기 위해 보안 관행을 준수해야 합니다.

|   #   | 설명                                                                                                                                                        | 레벨  | 역할  |
| :---: | --------------------------------------------------------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 3.4.1 | 모델 개발, 테스트 및 운영 환경이 물리적 또는 논리적으로 분리되어 있는지 확인하십시오. 이들은 공유 인프라가 없고, 별도의 접근 제어를 가지며, 격리된 데이터 저장소를 갖추고 있으며, 에이전트 오케스트레이션과 도구 또는 MCP 서버도 격리되어 있습니다.            |  1  | D/V |
| 3.4.2 | 모델 개발 산출물(하이퍼파라미터, 학습 스크립트, 구성 파일, 프롬프트 템플릿, 에이전트 정책/라우팅 그래프, 도구 또는 MCP 계약/스키마, 액션 카탈로그 또는 기능 허용 목록)이 버전 관리 시스템에 저장되고 학습에 사용되기 전에 동료 검토 승인을 요구하는지 확인하십시오. |  1  |  D  |
| 3.4.3 | 모델 학습 및 미세 조정이 egress 허용 목록을 사용하여 네트워크 접근이 통제된 격리된 환경에서 이루어지며, 프로덕션 도구나 MCP 리소스에 접근하지 않도록 확인하십시오.                                                         |  2  | D/V |
| 3.4.4 | 모델 개발에 사용되기 전에 RAG 인덱스, 툴 로그 및 미세 조정에 사용되는 에이전트 생성 데이터 등을 포함한 훈련 데이터 소스가 무결성 검사로 검증되고 문서화된 보관 연쇄와 함께 신뢰할 수 있는 소스를 통해 인증되었는지 확인하십시오.                       |  2  |  D  |

## C3.5 모델 은퇴 및 사용 중단

모델은 더 이상 필요하지 않거나 보안 문제가 확인되었을 때 안전하게 폐기되어야 합니다.

|   #   | 설명                                                                                                                                                                             | 레벨  | 역할  |
| :---: | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | :-: | :-: |
| 3.5.1 | 폐기된 모델 아티팩트(어댑터 및 안전/정책 모델 포함)가 안전한 암호화 소거를 사용하여 안전하게 삭제되었는지 확인합니다.                                                                                                            |  1  | D/V |
| 3.5.2 | 모델 폐기 이벤트가 타임스탬프와 행위자 신원, 모델 식별자(버전/다이제스트/서명), 그리고 추적 메타데이터(환경 및 사용하는 서비스/에이전트)와 함께 기록되는지 확인하며, 모델 서명이 폐기되고, 레지스트리/서빙 거부 목록과 로더 캐시 무효화가 에이전트가 폐기된 아티팩트를 로드하지 못하도록 방지하는지 검증합니다. |  2  |  V  |
