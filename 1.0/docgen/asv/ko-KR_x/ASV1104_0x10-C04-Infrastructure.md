# C4 인프라스트럭처, 구성 및 배포 보안

## 제어 목표

AI 인프라는 권한 상승, 공급망 변조, 측면 이동에 대비하여 보안 구성, 런타임 격리, 신뢰할 수 있는 배포 파이프라인, 그리고 포괄적인 모니터링을 통해 강화되어야 합니다. 검증되고 승인된 인프라 구성 요소만이 보안, 무결성, 감사를 보장하는 통제된 프로세스를 통해 운영 환경에 도달합니다.

---

## C4.1 런타임 환경 격리

OS 수준의 격리 원시 기능을 통해 컨테이너 탈출 및 권한 상승을 방지합니다.

|   #   | 설명                                                                                                        | 레벨  | 역할  |
| :---: | --------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 4.1.1 | 예를 들어 컨테이너의 경우 불필요한 Linux 권한을 제거하는 등 운영 체제에서 모든 AI 작업이 최소 권한으로 실행되는지 확인하세요.                               |  1  | D/V |
| 4.1.2 | 워크로드가 샌드박싱, seccomp 프로필, AppArmor, SELinux 또는 이와 유사한 악용 제한 기술로 보호되고 있으며, 구성도 적절한지 확인하십시오.                 |  1  | D/V |
| 4.1.3 | 워크로드가 읽기 전용 루트 파일 시스템에서 실행되며, 쓰기 가능한 마운트는 명확하게 정의되고 제한적인 옵션(e.g., noexec, nosuid, nodev)으로 강화되었는지 확인하십시오. |  2  | D/V |
| 4.1.4 | 런타임 모니터링이 권한 상승 및 컨테이너 탈출 행위를 탐지하고, 위반 프로세스를 자동으로 종료하는지 확인합니다.                                            |  2  | D/V |
| 4.1.5 | 높은 위험도의 AI 워크로드가 원격 증명이 성공한 후에만 하드웨어 분리 환경(예: TEE, 신뢰할 수 있는 하이퍼바이저, 또는 베어메탈 노드)에서 실행되는지 검증하십시오.           |  3  | D/V |

## C4.2 보안 빌드 및 배포 파이프라인

재현 가능한 빌드와 서명된 아티팩트를 통해 암호화 무결성과 공급망 보안을 보장하십시오.

|   #   | 설명                                                                           | 레벨  | 역할  |
| :---: | ---------------------------------------------------------------------------- | :-: | :-: |
| 4.2.1 | 빌드가 재현 가능하며 빌드 산출물에 대해 독립적으로 검증할 수 있는 서명된 출처 메타데이터를 적절하게 생성하는지 확인하십시오.       |  1  | D/V |
| 4.2.2 | 빌드가 소프트웨어 자재 명세서(SBOM)를 생성하고 배포 승인을 받기 전에 서명되었는지 확인하십시오.                     |  2  | D/V |
| 4.2.3 | 배포 시 빌드 산출물(예: 컨테이너 이미지)의 서명 및 출처 메타데이터가 검증되는지 확인하고, 검증되지 않은 산출물은 거부되도록 합니다. |  2  | D/V |

## C4.3 네트워크 보안 및 접근 제어

기본 거부 정책과 암호화된 통신을 사용하여 제로 트러스트 네트워킹을 구현하십시오.

|   #   | 설명                                                                                                                    | 레벨  | 역할  |
| :---: | --------------------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 4.3.1 | 네트워크 정책이 기본 거부 수신 및 발신을 강제하며, 필요한 서비스만 명시적으로 허용하는지 확인하십시오.                                                            |  1  | D/V |
| 4.3.2 | 관리자 접근 프로토콜(예: SSH, RDP) 및 클라우드 메타데이터 서비스에 대한 접근이 제한되며 강력한 인증을 요구하는지 확인하십시오.                                          |  1  | D/V |
| 4.3.3 | 출구 트래픽이 승인된 대상지로 제한되고 모든 요청이 기록되는지 확인하십시오.                                                                            |  2  | D/V |
| 4.3.4 | 인터서비스 통신이 인증서 검증과 정기적인 자동 갱신을 포함한 상호 TLS를 사용하는지 확인하십시오.                                                               |  2  | D/V |
| 4.3.5 | AI 워크로드와 환경(dev, test, prod)이 직접적인 인터넷 접근 없이, 공유된 IAM 역할, 보안 그룹 또는 환경 간 연결 없이 격리된 네트워크 구간(VPCs/VNets)에서 실행되는지 확인하십시오. |  2  | D/V |

## C4.4 비밀 및 암호키 관리

비밀과 암호화 키를 안전한 저장소, 자동 회전, 강력한 접근 제어로 보호하세요.

|   #   | 설명                                                                                                                   | 레벨  | 역할  |
| :---: | -------------------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 4.4.1 | 비밀 정보가 전담 비밀 관리 시스템에 저장되어 있으며, 휴지 상태 암호화가 적용되고 애플리케이션 작업 부하와 분리되어 있는지 확인하십시오.                                        |  1  | D/V |
| 4.4.2 | 암호화 키가 하드웨어 기반 모듈(예: HSM, 클라우드 KMS)에서 생성되고 저장되는지 확인하십시오.                                                             |  1  | D/V |
| 4.4.3 | 운영 비밀에 대한 접근이 강력한 인증을 요구하는지 확인하십시오.                                                                                  |  1  | D/V |
| 4.4.4 | 비밀 정보가 전용 비밀 관리 시스템을 통해 런타임에 애플리케이션에 배포되는지 확인하십시오. 비밀 정보는 소스 코드, 구성 파일, 빌드 아티팩트, 컨테이너 이미지 또는 환경 변수에 절대 포함되어서는 안 됩니다. |  1  | D/V |
| 4.4.5 | 비밀 값 회전이 자동화되어 있는지 확인하세요.                                                                                            |  2  | D/V |

## C4.5 AI 작업 부하 샌드박싱 및 검증

신뢰할 수 없는 AI 모델을 보안 샌드박스에서 격리하고, 신뢰 실행 환경(TEE) 및 기밀 컴퓨팅 기술을 사용하여 민감한 AI 작업 부하를 보호합니다.

|   #   | 설명                                                                                | 레벨  | 역할  |
| :---: | --------------------------------------------------------------------------------- | :-: | :-: |
| 4.5.1 | 외부 또는 신뢰할 수 없는 AI 모델이 격리된 샌드박스에서 실행되는지 확인하십시오.                                    |  1  | D/V |
| 4.5.2 | 샌드박스된 작업 부하는 기본적으로 아웃바운드 네트워크 연결이 없으며, 필요한 접근 권한은 명시적으로 정의되어 있는지 확인하십시오.          |  1  | D/V |
| 4.5.3 | 모델이나 워크로드 로딩 전에 워크로드 증명이 수행되어 신뢰할 수 있는 실행 환경에 대한 암호학적 증거가 확보되었는지 확인하십시오.          |  2  | D/V |
| 4.5.4 | 하드웨어 기반 격리, 메모리 암호화 및 무결성 보호를 제공하는 신뢰할 수 있는 실행 환경(TEE) 내에서 기밀 워크로드가 실행되는지 확인하십시오. |  3  | D/V |
| 4.5.5 | 암호화된 계산과 봉인된 모델 가중치, 보호된 실행을 통해 기밀 추론 서비스가 모델 추출을 방지하는지 확인하세요.                    |  3  | D/V |
| 4.5.6 | 신뢰할 수 있는 실행 환경의 오케스트레이션이 수명 주기 관리, 원격 증명, 암호화된 통신 채널을 포함하는지 확인하십시오.               |  3  | D/V |
| 4.5.7 | 보안 다자간 계산(SMPC)이 개별 데이터셋이나 모델 매개변수를 노출하지 않고 협업 AI 학습을 가능하게 하는지 검증하세요.             |  3  | D/V |

## C4.6 AI 인프라 자원 관리, 백업 및 복구

쿼터와 모니터링을 통해 자원 소모 공격을 방지하고 공정한 자원 할당을 보장합니다. 안전한 백업, 테스트된 복구 절차 및 재해 복구 기능을 통해 인프라 복원력을 유지합니다.

|   #   | 설명                                                                                                                               | 레벨  | 역할  |
| :---: | -------------------------------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 4.6.1 | 워크로드의 리소스 소비가 Kubernetes ResourceQuotas 또는 유사한 도구를 사용하여 서비스 거부 공격을 완화할 수 있도록 적절하게 제한되는지 검증하십시오.                                  |  2  | D/V |
| 4.6.2 | 정의된 CPU, 메모리 또는 요청 임계값을 초과하면 리소스 고갈이 자동 보호 기능(예: 속도 제한 또는 작업 부하 분리)을 트리거하는지 확인하십시오.                                              |  2  | D/V |
| 4.6.3 | 백업 시스템이 별도의 자격 증명을 사용하는 격리된 네트워크에서 운영되는지 확인하고, 저장 시스템이 에어갭 네트워크에서 운영되거나 무단 수정 방지를 위해 WORM(한 번 쓰기 후 여러 번 읽기) 보호 기능을 구현하는지 검증하십시오. |  2  | D/V |

## C4.7 AI 하드웨어 보안

GPU, TPU 및 특수 AI 가속기를 포함한 AI 전용 하드웨어 구성 요소를 보호하십시오.

|   #   | 설명                                                                                                                          | 레벨  | 역할  |
| :---: | --------------------------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 4.7.1 | 워크로드 실행 전에 AI 가속기 무결성이 하드웨어 기반 증명 메커니즘(예: TPM, DRTM 또는 이에 상응하는 방법)을 사용하여 검증되었는지 확인하십시오.                                     |  2  | D/V |
| 4.7.2 | 가속기(GPU) 메모리가 작업 간에 파티셔닝 메커니즘을 통해 분리되고 작업 간 메모리 정리가 이루어지는지 확인하십시오.                                                          |  2  | D/V |
| 4.7.3 | 하드웨어 보안 모듈(HSM)이 AI 모델 가중치와 암호화 키를 FIPS 140-3 Level 3 또는 Common Criteria EAL4+ 인증으로 보호하는지 확인하십시오.                           |  3  | D/V |
| 4.7.4 | 가속기 펌웨어(GPU/TPU/NPUs)가 버전 고정(version-pinned), 서명(signing), 부팅 시 증명(attested)된 상태인지 확인하십시오; 서명되지 않았거나 디버그(debug) 펌웨어는 차단됩니다. |  2  | D/V |
| 4.7.5 | VRAM 및 온-패키지 메모리가 작업/테넌트 간에 초기화되는지 확인하고, 장치 재설정 정책이 테넌트 간 데이터 잔류를 방지하는지 검증하십시오.                                             |  2  | D/V |
| 4.7.6 | 파티셔닝/격리 기능(예: MIG/VM 파티셔닝)이 각 테넌트별로 적용되어 파티션 간 동료 간 메모리 접근을 방지하는지 확인하십시오.                                                   |  2  | D/V |
| 4.7.7 | 가속기 인터커넥트(NVLink/PCIe/InfiniBand/RDMA/NCCL)가 승인된 토폴로지와 인증된 엔드포인트로 제한되어 있는지 확인하십시오; 평문 크로스 테넌트 링크는 허용되지 않습니다.                |  3  | D/V |
| 4.7.8 | 가속기 원격 측정 데이터(전력, 온도, ECC, 성능 카운터)가 SIEM/OTel로 내보내지고 있으며, 부채널 또는 은밀 채널을 나타내는 이상 징후에 대해 경고가 발생하는지 확인하십시오.                    |  3  |  D  |

## C4.8 에지 및 분산 AI 보안

엣지 컴퓨팅, 연합 학습, 다중 사이트 아키텍처를 포함한 보안 분산 AI 배포.

|   #   | 설명                                                                                                                                                          | 레벨  | 역할  |
| :---: | ----------------------------------------------------------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 4.8.1 | 에지 AI 장치가 상호 TLS를 사용하여 중앙 인프라에 인증하는지 확인하십시오.                                                                                                                |  2  | D/V |
| 4.8.2 | 에지 디바이스가 검증된 서명과 롤백 보호 기능을 갖춘 보안 부팅을 구현하여 펌웨어 다운그레이드 공격을 방지하는지 확인하십시오.                                                                                      |  2  | D/V |
| 4.8.3 | 분산 AI 조정이 참가자 검증 및 악의적인 노드 감지를 포함한 비잔틴 결함 허용 합의 메커니즘을 사용하는지 확인하십시오.                                                                                         |  3  | D/V |
| 4.8.4 | 에지-투-클라우드 통신이 대역폭 제한, 데이터 압축 및 암호화된 로컬 저장소를 통한 보안 오프라인 운영을 지원하는지 확인하십시오.                                                                                    |  3  | D/V |
| 4.8.5 | 모바일 또는 엣지 추론 애플리케이션이 수정된 바이너리, 재패키지된 앱, 또는 연결된 계측 프레임워크를 감지하고 차단하는 플랫폼 수준의 변조 방지 보호(예: 코드 서명, 검증 부팅, 런타임 자기 무결성 검사)를 구현하는지 확인하십시오.                          |  3  | D/V |
| 4.8.6 | 엣지 또는 모바일 장치에 배포된 모델이 패키징 과정에서 암호학적으로 서명되었는지 확인하고, 장치 내 런타임이 로드 또는 추론 전에 이러한 서명 또는 체크섬을 검증하는지 확인해야 합니다; 검증되지 않았거나 변경된 모델은 반드시 거부되어야 합니다.                    |  3  | D/V |
| 4.8.7 | 온-디바이스 추론 런타임이 프로세스, 메모리 및 파일 접근 격리를 강제하여 모델 덤핑, 디버깅 또는 중간 임베딩과 활성화 값 추출을 방지하는지 확인하십시오.                                                                     |  3  | D/V |
| 4.8.8 | 모델 가중치 및 로컬에 저장된 민감한 매개변수가 하드웨어 기반 키 저장소 또는 보안 인클레이브(예: Android Keystore, iOS Secure Enclave, TPM/TEE)를 사용하여 암호화되어 있으며, 키가 사용자 공간에서 접근할 수 없도록 되어 있는지 확인합니다. |  3  | D/V |
| 4.8.9 | 모바일, IoT 또는 임베디드 애플리케이션에 포함된 모델이 저장 시 암호화 또는 난독화되어 있으며, 신뢰할 수 있는 런타임 또는 보안 인클레이브 내에서만 복호화되어 앱 패키지나 파일 시스템에서 직접 추출되는 것을 방지하는지 확인합니다.                         |  3  | D/V |
