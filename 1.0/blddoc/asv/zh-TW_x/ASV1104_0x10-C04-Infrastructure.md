# C4 基礎設施、配置與部署安全

## 控制目標

AI 基礎設施必須透過安全配置、運行時隔離、可信部署管線以及全面監控來加強防護，以防止權限提升、供應鏈篡改和橫向移動。只有經授權且經驗證的基礎設施設備和配置，才能透過維持安全性、完整性和可審計性的受控流程進入生產環境。

核心安全目標：僅經過密碼學簽名、漏洞掃描的基礎設施組件，通過自動化驗證管線進入生產環境，該管線強制執行安全政策並維護不可變的審計追蹤。

## C4.1 執行環境隔離

通過內核級隔離原語和強制訪問控制，防止容器逃逸和權限提升。

|   #   | 描述                                                                     | 等級  | 角色  |
| :---: | ---------------------------------------------------------------------- | :-: | :-: |
| 4.1.1 | 確認所有 AI 容器均剝除除 CAP_SETUID、CAP_SETGID 及安全基線中明確要求的能力以外的所有 Linux 能力。      |  1  | D/V |
| 4.1.2 | 驗證 seccomp 配置檔是否封鎖所有系統呼叫，僅允許預先核准的白名單中的系統呼叫，違規時終止容器並產生安全警報。             |  1  | D/V |
| 4.1.3 | 驗證 AI 工作負載是否以唯讀根檔案系統運行，使用 tmpfs 作為臨時資料，並對持久化資料使用帶有 noexec 掛載選項的命名卷。    |  2  | D/V |
| 4.1.4 | 驗證基於 eBPF 的執行時監控（Falco、Tetragon 或同等工具）是否能檢測特權提升嘗試，並在組織回應時間要求內自動終止違規進程。 |  2  | D/V |
| 4.1.5 | 驗證高風險的 AI 工作負載是否在硬體隔離環境中執行（Intel TXT、AMD SVM 或專用裸機節點），並進行驗證證明。         |  3  | D/V |

## C4.2 安全建置與部署流程

透過可重現構建和簽名工件，確保密碼學完整性及供應鏈安全。

|   #   | 描述                                                                                           | 等級  | 角色  |
| :---: | -------------------------------------------------------------------------------------------- | :-: | :-: |
| 4.2.1 | 確認基礎設施即程式碼在每次提交時皆使用工具（如 tfsec、Checkov 或 Terrascan）進行掃描，並且對於具有 CRITICAL 或 HIGH 嚴重性問題的發現，阻止合併。 |  1  | D/V |
| 4.2.2 | 驗證容器構建是否可重現，確保各次構建產生相同的 SHA256 雜湊值，並使用 Sigstore 簽署，生成符合 SLSA Level 3 水準的起源證明。                |  1  | D/V |
| 4.2.3 | 驗證容器映像檔在推送至註冊中心前，內嵌 CycloneDX 或 SPDX SBOM 並使用 Cosign 簽署，未簽署的映像檔在部署時被拒絕。                      |  2  | D/V |
| 4.2.4 | 驗證 CI/CD 管道使用來自 HashiCorp Vault、AWS IAM 角色或 Azure 管理身份的 OIDC 令牌，其有效期限不得超過組織安全政策限制。           |  2  | D/V |
| 4.2.5 | 在容器執行前的部署過程中，驗證 Cosign 簽名和 SLSA 溯源，並確保驗證錯誤會導致部署失敗。                                           |  2  | D/V |
| 4.2.6 | 確認構建環境在臨時容器或虛擬機器（VM）中運行，且無持久性存儲並與生產虛擬私有雲（VPC）實現網路隔離。                                         |  2  | D/V |

## C4.3 網路安全與存取控制

實施零信任網路，採用預設拒絕政策與加密通訊。

|   #   | 描述                                                                                   | 等級  | 角色  |
| :---: | ------------------------------------------------------------------------------------ | :-: | :-: |
| 4.3.1 | 驗證 Kubernetes NetworkPolicies 或任何等效機制是否實作了預設拒絕的進入/外發流量，並針對所需連接埠（443、8080 等）設有明確允許規則。 |  1  | D/V |
| 4.3.2 | 請確認 SSH（端口 22）、RDP（端口 3389）和雲端元資料端點（169.254.169.254）已被阻擋或需要基於憑證的驗證。                  |  1  | D/V |
| 4.3.3 | 驗證出站流量是否透過 HTTP/HTTPS 代理（Squid、Istio 或雲端 NAT 閘道）進行過濾，且具備網域允許清單，並記錄被阻擋的請求。            |  2  | D/V |
| 4.3.4 | 驗證服務間通訊使用相互 TLS，且憑證依組織政策輪換，並強制執行憑證驗證（不得使用跳過驗證標誌）。                                    |  2  | D/V |
| 4.3.5 | 確認 AI 基礎設施運行在專用 VPCs/VNets 中，且沒有直接的網際網路存取，僅透過 NAT 網關或堡壘主機進行通訊。                       |  2  | D/V |

## C4.4 秘密與密鑰管理

透過硬體支援的儲存和自動輪替，以及零信任存取，保護認證資訊。

|   #   | 描述                                                                                                          | 等級  | 角色  |
| :---: | ----------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 4.4.1 | 驗證秘密是否儲存在 HashiCorp Vault、AWS Secrets Manager、Azure Key Vault 或 Google Secret Manager 中，並使用 AES-256 進行靜態加密。 |  1  | D/V |
| 4.4.2 | 驗證密鑰在符合 FIPS 140-2 Level 2 的硬體安全模組（HSM）（如 AWS CloudHSM、Azure Dedicated HSM）中生成，並依照組織的密碼政策進行密鑰輪換。            |  1  | D/V |
| 4.4.3 | 驗證密鑰輪替是否自動化，並確保零停機部署及因人事變動或安全事件立即啟動輪替。                                                                      |  2  | D/V |
| 4.4.4 | 確認容器映像檔已使用工具（GitLeaks、TruffleHog 或 detect-secrets）進行掃描，並阻止包含 API 金鑰、密碼或憑證的建置。                               |  2  | D/V |
| 4.4.5 | 驗證生產環境的秘密存取是否需要使用硬體令牌（如 YubiKey、FIDO2）的多因素身份驗證（MFA），並且是否由不可更改的審計日誌記錄，包含用戶身份和時間戳。                            |  2  | D/V |
| 4.4.6 | 請確認機密資訊是透過 Kubernetes 機密、掛載的卷或初始化容器注入，並確保機密資訊絕不會被嵌入在環境變數或映像檔中。                                              |  2  | D/V |

## C4.5 AI 工作負載沙盒化與驗證

在安全的沙盒中隔離不受信任的 AI 模型，並進行全面的行為分析。

|   #   | 描述                                                                                                                         | 等級  | 角色  |
| :---: | -------------------------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 4.5.1 | 驗證外部 AI 模型是否在 gVisor、microVM（例如 Firecracker、CrossVM）或 Docker 容器中執行，並且使用 --security-opt=no-new-privileges 和 --read-only 標誌。 |  1  | D/V |
| 4.5.2 | 確認沙箱環境沒有網路連接（--network=none）或僅允許本地主機存取，並且所有外部請求皆由 iptables 規則封鎖。                                                           |  1  | D/V |
| 4.5.3 | 驗證 AI 模型驗證過程包含具組織定義測試覆蓋範圍與行為分析的自動化紅隊測試，以偵測後門。                                                                              |  2  | D/V |
| 4.5.4 | 確認在 AI 模型投入生產之前，其沙盒結果已由授權的安全人員進行密碼學簽名，並存儲在不可變更的審計日誌中。                                                                      |  2  | D/V |
| 4.5.5 | 確認沙盒環境在評估之間從黃金映像中銷毀並重新創建，並進行完整的檔案系統和記憶體清理。                                                                                 |  2  | D/V |
