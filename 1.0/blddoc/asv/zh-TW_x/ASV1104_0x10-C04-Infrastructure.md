# C4 基礎設施、配置與部署安全

## 控制目標

人工智慧基礎設施必須通過安全配置、運行時隔離、可信部署管道和全面監控來加強防禦，防止特權提升、供應鏈篡改和橫向移動。只有經過授權和驗證的基礎設施組件和配置，才能通過受控流程進入生產環境，確保安全性、完整性和可審計性。

核心安全目標：僅通過自動化驗證流程，包含密碼學簽名及漏洞掃描的基礎設施元件，才能進入生產環境，此流程強制執行安全政策並維持不可變更的稽核軌跡。

## C4.1 執行時環境隔離

透過核心級隔離原語和強制性存取控管，防止容器逃逸及權限提升。

|   #   | 描述                                                                     | 層級  | 角色  |
| :---: | ---------------------------------------------------------------------- | :-: | :-: |
| 4.1.1 | 驗證所有 AI 容器是否丟棄除了 CAP_SETUID、CAP_SETGID 以及安全基線中明確要求的權限之外的所有 Linux 權限。   |  1  | D/V |
| 4.1.2 | 驗證 seccomp 配置檔是否阻止所有系統調用，僅允許預先核准的允許清單中的系統調用，違規時終止容器並產生安全警報。            |  1  | D/V |
| 4.1.3 | 驗證 AI 工作負載是否以唯讀根檔案系統運行，暫存資料使用 tmpfs，持久資料使用具 noexec 掛載選項的命名卷。           |  2  | D/V |
| 4.1.4 | 驗證基於 eBPF 的運行時監控（Falco、Tetragon 或同等工具）是否能偵測權限升級嘗試，並在組織回應時間要求內自動終止違規進程。 |  2  | D/V |
| 4.1.5 | 驗證高風險 AI 工作負載是否在硬體隔離環境中執行（Intel TXT、AMD SVM 或專用裸機節點），並進行驗證背書。          |  3  | D/V |

## C4.2 安全構建與部署流程

透過可重現構建和簽名工件，確保密碼學完整性與供應鏈安全。

|   #   | 描述                                                                                          | 層級  | 角色  |
| :---: | ------------------------------------------------------------------------------------------- | :-: | :-: |
| 4.2.1 | 確保在每次提交時，基礎設施即程式碼都使用工具（tfsec、Checkov 或 Terrascan）進行掃描，並且對於具有 CRITICAL 或 HIGH 嚴重性發現的提交，阻止合併。 |  1  | D/V |
| 4.2.2 | 驗證容器構建在多次構建中具備可重現性，且具有相同的 SHA256 雜湊值，並生成經由 Sigstore 簽署的 SLSA 級別 3 來源聲明。                     |  1  | D/V |
| 4.2.3 | 驗證容器映像檔在推送至映像檔註冊中心前嵌入了 CycloneDX 或 SPDX SBOM，並使用 Cosign 進行簽署，未簽署的映像檔在部署時將被拒絕。               |  2  | D/V |
| 4.2.4 | 驗證 CI/CD 管道是否使用 HashiCorp Vault、AWS IAM 角色或 Azure 管理身分識別的 OIDC 令牌，且其有效期限不超過組織安全政策的限制。       |  2  | D/V |
| 4.2.5 | 在容器執行前的部署過程中，驗證 Cosign 簽名和 SLSA 來源是否被正確驗證，並確保驗證錯誤會導致部署失敗。                                   |  2  | D/V |
| 4.2.6 | 驗證建置環境是否在臨時容器或虛擬機器中運行，且無持久性儲存以及與生產 VPC 隔離的網路環境。                                             |  2  | D/V |

## C4.3 網路安全與存取控制

實施零信任網路，採用預設拒絕策略及加密通訊。

|   #   | 描述                                                                                    | 層級  | 角色  |
| :---: | ------------------------------------------------------------------------------------- | :-: | :-: |
| 4.3.1 | 確認 Kubernetes NetworkPolicies 或任何同等機制是否實作了預設拒絕的入口/出口流量規則，並對必要的埠（443、8080 等）設定明確的允許規則。 |  1  | D/V |
| 4.3.2 | 確認 SSH（連接埠 22）、RDP（連接埠 3389）和雲端元資料端點（169.254.169.254）是否被封鎖或需要基於憑證的驗證。                 |  1  | D/V |
| 4.3.3 | 驗證出口流量是否通過帶有網域允許名單的 HTTP/HTTPS 代理（如 Squid、Istio 或雲端 NAT 閘道器）進行過濾，並記錄被阻擋的請求。           |  2  | D/V |
| 4.3.4 | 驗證服務間通訊是否使用相互 TLS，並依照組織政策進行憑證輪替，且強制執行憑證驗證（不可使用跳過驗證標誌）。                                |  2  | D/V |
| 4.3.5 | 驗證 AI 基礎設施是否運行於專用的 VPC/VNet 中，沒有直接的網際網路存取，且僅透過 NAT 網關或堡壘主機進行通訊。                       |  2  | D/V |

## C4.4 密鑰與加密金鑰管理

透過硬體支援的儲存和自動輪替，以零信任存取保護憑證。

|   #   | 描述                                                                                                          | 層級  | 角色  |
| :---: | ----------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 4.4.1 | 驗證機密是否以靜態加密方式（使用 AES-256）儲存在 HashiCorp Vault、AWS Secrets Manager、Azure Key Vault 或 Google Secret Manager 中。 |  1  | D/V |
| 4.4.2 | 驗證加密金鑰是否在符合 FIPS 140-2 Level 2 標準的 HSM（AWS CloudHSM、Azure Dedicated HSM）中生成，並根據組織的加密政策進行金鑰輪替。               |  1  | D/V |
| 4.4.3 | 驗證秘密輪換是否實現自動化，包括零停機部署，以及是否在人員變動或安全事件發生時觸發即時輪換。                                                              |  2  | D/V |
| 4.4.4 | 驗證容器映像檔是否使用工具（GitLeaks、TruffleHog 或 detect-secrets）進行掃描，阻止包含 API 金鑰、密碼或憑證的建置。                               |  2  | D/V |
| 4.4.5 | 驗證生產環境的祕密存取需要使用硬體憑證（YubiKey、FIDO2）的多重身份驗證（MFA），並且該存取行為必須由包含使用者身分和時間戳記的不可更改稽核日誌記錄。                           |  2  | D/V |
| 4.4.6 | 確認秘鑰是通過 Kubernetes 秘鑰、掛載的卷或初始化容器注入，並確保秘鑰絕不嵌入環境變數或映像中。                                                       |  2  | D/V |

## C4.5 AI 工作負載沙箱測試與驗證

|   #   | 描述                                                                                                                     | 層級  | 角色  |
| :---: | ---------------------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 4.5.1 | 驗證外部 AI 模型是否在 gVisor、微型虛擬機（如 Firecracker、CrossVM）或啟用了 --security-opt=no-new-privileges 和 --read-only 旗標的 Docker 容器中執行。 |  1  | D/V |
| 4.5.2 | 確認沙盒環境沒有網路連線（--network=none）或僅限本地主機存取，並且所有外部請求皆被 iptables 規則阻擋。                                                        |  1  | D/V |
| 4.5.3 | 驗證 AI 模型驗證包含具組織定義的測試範圍及行為分析的自動化紅隊測試，以檢測後門。                                                                             |  2  | D/V |
| 4.5.4 | 確認在 AI 模型升級至生產環境之前，其沙盒結果已由授權的安全人員進行加密簽名，並存儲在不可變更的審計日誌中。                                                                |  2  | D/V |
| 4.5.5 | 驗證沙盒環境在評估之間已從黃金映像重新創建並銷毀，確保完成文件系統和記憶體的清理。                                                                              |  2  | D/V |
