# C3 모델 수명 주기 관리 및 변경 제어

## 제어 목표

AI 시스템은 무단 또는 안전하지 않은 모델 수정이 프로덕션에 반영되는 것을 방지하는 변경 통제 프로세스를 구현해야 합니다. 이 통제는 개발부터 배포, 폐기까지 전체 수명 주기 동안 모델의 무결성을 보장하며, 이를 통해 신속한 사고 대응이 가능하고 모든 변경에 대한 책임을 유지할 수 있습니다.

핵심 보안 목표: 무결성, 추적 가능성 및 복구 가능성을 유지하는 통제된 프로세스를 사용하여 승인되고 검증된 모델만이 생산 단계에 도달하도록 합니다.

---

## C3.1 모델 승인 및 무결성

검증된 무결성을 가진 권한 있는 모델만이 운영 환경에 도달합니다.

|   #   | 설명                                                                                              | 레벨  | 역할  |
| :---: | ----------------------------------------------------------------------------------------------- | :-: | :-: |
| 3.1.1 | 배포 전에 모든 모델 아티팩트(가중치, 구성, 토크나이저)가 권한 있는 엔터티에 의해 암호화 서명되었는지 확인하십시오.                              |  1  | D/V |
| 3.1.2 | 모델 무결성이 배포 시점에 검증되고 서명 검증 실패 시 모델 로딩이 차단되는지 확인하십시오.                                             |  1  | D/V |
| 3.1.3 | 모델 출처 기록에 승인 기관의 신원, 학습 데이터 체크섬, 통과/실패 상태가 포함된 검증 테스트 결과, 생성 타임스탬프가 포함되어 있는지 확인하십시오.            |  2  | D/V |
| 3.1.4 | 모든 모델 아티팩트가 의미적 버전 관리(MAJOR.MINOR.PATCH)를 사용하며, 각 버전 구성 요소가 증가하는 시점을 명시한 문서화된 기준을 사용하는지 검증하십시오. |  2  | D/V |
| 3.1.5 | 종속성 추적이 모든 소비 시스템을 신속하게 식별할 수 있는 실시간 재고를 유지하는지 확인하십시오.                                          |  2  |  V  |

---

## C3.2 모델 검증 및 테스트

모델은 배포 전에 정의된 보안 및 안전성 검증을 통과해야 합니다.

|   #   | 설명                                                                           | 레벨  | 역할  |
| :---: | ---------------------------------------------------------------------------- | :-: | :-: |
| 3.2.1 | 모델이 배포 전에 입력 검증, 출력 정화, 사전 합의된 조직의 합격/불합격 기준을 포함한 자동화된 보안 테스트를 거치는지 검증하십시오.  |  1  | D/V |
| 3.2.2 | 검증 실패가 명시된 사전 지정 권한자에 의한 명확한 재승인과 문서화된 비즈니스 정당성 없이 모델 배포를 자동으로 차단하는지 확인하십시오. |  1  | D/V |
| 3.2.3 | 테스트 결과가 암호화적으로 서명되고 검증 중인 특정 모델 버전 해시와 불변하게 연결되어 있는지 확인하십시오.                 |  2  |  V  |
| 3.2.4 | 비상 배포는 사전에 합의된 시간 내에 미리 지정된 보안 권한자로부터 문서화된 보안 위험 평가와 승인을 받아야 함을 확인하십시오.      |  2  | D/V |

---

## C3.3 통제된 배포 및 롤백

모델 배포는 통제되고, 모니터링되며, 되돌릴 수 있어야 합니다.

|   #   | 설명                                                                                                           | 레벨  | 역할  |
| :---: | ------------------------------------------------------------------------------------------------------------ | :-: | :-: |
| 3.3.1 | 프로덕션 배포가 사전에 합의된 오류율, 지연 시간 임계값 또는 보안 경고 기준에 기반한 자동 롤백 트리거와 함께 점진적 롤아웃 메커니즘(카나리 배포, 블루-그린 배포)을 구현하는지 확인하십시오. |  1  |  D  |
| 3.3.2 | 롤백 기능이 미리 정의된 조직 내 시간 창 내에서 전체 모델 상태(가중치, 구성, 종속성)를 원자적으로 복원하는지 확인하십시오.                                      |  1  | D/V |
| 3.3.3 | 배포 프로세스가 모델 활성화 전에 암호화 서명을 검증하고 무결성 체크섬을 계산하며, 불일치가 있을 경우 배포를 실패하도록 확인하십시오.                                  |  2  | D/V |
| 3.3.4 | 비상 모델 종료 기능이 자동 회로 차단기 또는 수동 종료 스위치를 통해 사전에 정의된 응답 시간 내에 모델 엔드포인트를 비활성화할 수 있음을 확인하십시오.                       |  2  | D/V |
| 3.3.5 | 조직 정책에 따라 롤백 아티팩트(이전 모델 버전, 구성, 종속성)가 사건 대응을 위해 변경 불가능한 저장소에 보관되는지 확인합니다.                                    |  2  |  V  |

## C3.4 변경 책임 및 감사

모든 모델 수명주기 변경 사항은 추적 가능하고 감사할 수 있어야 합니다.

|   #   | 설명                                                                                                     | 레벨  | 역할  |
| :---: | ------------------------------------------------------------------------------------------------------ | :-: | :-: |
| 3.4.1 | 모든 모델 변경(배포, 구성, 폐기)이 타임스탬프, 인증된 행위자 신원, 변경 유형, 이전/이후 상태를 포함하는 불변의 감사 기록을 생성하는지 확인하십시오.                |  1  |  V  |
| 3.4.2 | 감사 로그 접근이 적절한 권한을 요구하며 모든 접근 시도가 사용자 신원과 타임스탬프와 함께 기록되는지 확인하십시오.                                       |  2  | D/V |
| 3.4.3 | 배포 전에 지정된 검토자의 필수 코드 리뷰 및 승인을 통해 프롬프트 템플릿과 시스템 메시지가 git 저장소에서 버전 관리되고 있는지 확인하십시오.                      |  2  | D/V |
| 3.4.4 | 감사 기록에 모델 해시, 구성 스냅샷, 의존성 버전 등의 충분한 세부 정보가 포함되어 있어 보유 기간 내 임의의 타임스탬프에 대해 모델 상태를 완전히 재구성할 수 있는지 확인하십시오. |  2  |  V  |

## C3.5 보안 개발 관행

모델 개발 및 학습 프로세스는 침해를 방지하기 위해 보안 관행을 따라야 합니다.

|   #   | 설명                                                                                                        | 레벨  | 역할  |
| :---: | --------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 3.5.1 | 모델 개발, 테스트 및 운영 환경이 물리적 또는 논리적으로 분리되어 있는지 확인하십시오. 이들은 공유 인프라가 없고, 별도의 액세스 제어를 가지며, 데이터 저장소가 격리되어 있어야 합니다. |  1  |  D  |
| 3.5.2 | 모델 훈련 및 미세 조정이 네트워크 접근이 통제된 격리된 환경에서 이루어지는지 확인하십시오.                                                       |  1  |  D  |
| 3.5.3 | 모델 개발에 사용하기 전에 훈련 데이터 소스가 무결성 검사로 검증되고, 문서화된 관리 이력을 통해 신뢰할 수 있는 출처에서 인증되었는지 확인하십시오.                       |  1  | D/V |
| 3.5.4 | 모델 개발 산출물(하이퍼파라미터, 학습 스크립트, 구성 파일)이 버전 관리 시스템에 저장되고 학습에 사용되기 전에 동료 검토 승인을 받아야 하는지 확인하십시오.                 |  2  |  D  |

## C3.6 모델 폐기 및 사용 중단

모델은 더 이상 필요하지 않거나 보안 문제가 발견될 경우 안전하게 폐기되어야 합니다.

|   #   | 설명                                                                                              | 레벨  | 역할  |
| :---: | ----------------------------------------------------------------------------------------------- | :-: | :-: |
| 3.6.1 | 모델 폐기 프로세스가 의존성 그래프를 자동으로 스캔하고, 모든 소비 시스템을 식별하며, 폐기 전에 사전 합의된 통보 기간을 제공하는지 확인하십시오.              |  1  |  D  |
| 3.6.2 | 폐기된 모델 인공물을 문서화된 데이터 보존 정책에 따라 암호화 소거 또는 다중 덮어쓰기를 이용해 안전하게 지우고, 확인된 파기 증명서를 통해 이를 검증했는지 확인하십시오. |  1  | D/V |
| 3.6.3 | 모델 은퇴 이벤트가 타임스탬프와 행위자 신원과 함께 기록되었는지, 그리고 재사용을 방지하기 위해 모델 서명이 취소되었는지 확인하십시오.                     |  2  |  V  |
| 3.6.4 | 긴급 상황 모델 은퇴가 중요 보안 취약점이 발견될 경우 자동 킬 스위치를 통해 사전에 정해진 긴급 대응 시간 내에 모델 접근을 비활성화할 수 있는지 확인하십시오.      |  2  | D/V |
