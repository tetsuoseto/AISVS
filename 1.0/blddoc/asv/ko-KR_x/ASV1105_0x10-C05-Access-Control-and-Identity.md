# AI 구성 요소 및 사용자용 C5 액세스 제어 및 신원 관리

## 통제 목표

AI 시스템에 대한 효과적인 접근 제어는 강력한 신원 관리, 상황 인식 권한 부여, 그리고 제로 트러스트 원칙에 따른 실행 시점 강화가 필요합니다. 이러한 제어는 인간, 서비스, 자율 에이전트가 명시적으로 허용된 범위 내에서만 모델, 데이터, 컴퓨팅 자원과 상호작용하도록 보장하며, 지속적인 검증과 감사 기능을 제공합니다.

---

## C5.1 신원 관리 및 인증

권한 있는 작업에 대해 다중 인증을 사용하여 모든 개체에 대해 암호화로 보장된 신원을 구축하십시오.

|   #   | 설명                                                                                                                  | 레벨  | 역할  |
| :---: | ------------------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 5.1.1 | 모든 인간 사용자와 서비스 주체가 고유한 신원-토큰 매핑을 사용하여 중앙 집중식 기업 아이덴티티 제공자(IdP)를 통해 OIDC/SAML 프로토콜로 인증하는지 확인하십시오(공유 계정 또는 자격 증명 없음). |  1  | D/V |
| 5.1.2 | 고위험 작업(모델 배포, 가중치 내보내기, 학습 데이터 접근, 운영 구성 변경)이 다단계 인증 또는 세션 재검증이 포함된 단계별 인증을 요구하는지 확인하십시오.                           |  1  | D/V |
| 5.1.3 | 새로운 책임자가 운영 시스템 접근 권한을 받기 전에 NIST 800-63-3 IAL-2 또는 동등한 기준에 부합하는 신원 인증 절차를 거쳤는지 확인하십시오.                             |  2  |  D  |
| 5.1.4 | 접근 검토가 분기별로 수행되며 휴면 계정 자동 감지, 자격 증명 교체 시행, 권한 해제 워크플로우가 포함되어 있는지 확인하십시오.                                            |  2  |  V  |
| 5.1.5 | 연합 AI 에이전트가 최대 수명이 24시간이며 출처에 대한 암호학적 증거를 포함하는 서명된 JWT 어설션을 통해 인증하는지 검증하십시오.                                        |  3  | D/V |

## C5.2 리소스 권한 부여 및 최소 권한 원칙

모든 AI 리소스에 대해 명시적인 권한 모델과 감사 추적이 포함된 세밀한 접근 제어를 구현합니다.

|   #   | 설명                                                                                                         | 레벨  | 역할  |
| :---: | ---------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 5.2.1 | 모든 AI 리소스(데이터셋, 모델, 엔드포인트, 벡터 컬렉션, 임베딩 인덱스, 컴퓨트 인스턴스)가 명시적인 허용 목록과 기본 거부 정책을 통해 역할 기반 접근 제어를 시행하는지 확인하십시오. |  1  | D/V |
| 5.2.2 | 서비스 계정에 대해 최소 권한 원칙이 기본적으로 적용되어 읽기 전용 권한부터 시작하며, 쓰기 권한 부여 시에는 문서화된 업무상의 정당한 사유가 요구되는지 확인하십시오.              |  1  | D/V |
| 5.2.3 | 모든 접근 제어 수정이 승인된 변경 요청과 연결되어 있으며, 타임스탬프, 수행자 신원, 리소스 식별자 및 권한 델타와 함께 불변으로 기록되었는지 검증합니다.                    |  1  |  V  |
| 5.2.4 | 데이터 분류 라벨(PII, PHI, 수출 통제, 독점)이 파생 리소스(임베딩, 프롬프트 캐시, 모델 출력)에 자동으로 전파되고 일관된 정책 적용이 이루어지는지 검증하십시오.           |  2  |  D  |
| 5.2.5 | 무단 접근 시도와 권한 상승 이벤트가 5분 이내에 컨텍스트 메타데이터와 함께 SIEM 시스템에 실시간 경고를 발생시키는지 확인하십시오.                                |  2  | D/V |

## C5.3 동적 정책 평가

감사 기능을 포함한 컨텍스트 인식 권한 부여 결정을 위해 속성 기반 접근 제어(ABAC) 엔진을 배포하세요.

|   #   | 설명                                                                                             | 레벨  | 역할  |
| :---: | ---------------------------------------------------------------------------------------------- | :-: | :-: |
| 5.3.1 | 승인 결정이 인증된 API를 통해 접근 가능하며 암호화 무결성 보호가 적용된 전용 정책 엔진(OPA, Cedar 또는 동등한 시스템)으로 외부화되어 있는지 검증하십시오. |  1  | D/V |
| 5.3.2 | 정책이 실행 시점에 사용자 보안 등급, 자원 민감도 분류, 요청 맥락, 테넌트 격리, 시간적 제약 등 동적 속성을 평가하는지 확인하십시오.                  |  1  | D/V |
| 5.3.3 | 정책 정의가 버전 관리되고, 동료 검토를 거치며, CI/CD 파이프라인에서 자동화된 테스트를 통해 검증된 후에 생산 배포되는지 확인하십시오.                 |  2  |  D  |
| 5.3.4 | 정책 평가 결과에 구조화된 결정 근거가 포함되어 있으며, 상관 관계 분석 및 규정 준수 보고를 위해 SIEM 시스템으로 전송되는지 확인하십시오.               |  2  |  V  |
| 5.3.5 | 정책 캐시의 수명 시간(TTL) 값이 고감도 리소스에 대해 5분을 초과하지 않고, 캐시 무효화 기능이 있는 표준 리소스에 대해 1시간을 초과하지 않는지 확인하십시오.   |  3  | D/V |

## C5.4 쿼리 실행 시 보안 적용

필수 필터링 및 행 수준 보안 정책과 함께 데이터베이스 계층 보안 제어를 구현하십시오.

|   #   | 설명                                                                                                                   | 레벨  | 역할  |
| :---: | -------------------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 5.4.1 | 모든 벡터 데이터베이스 및 SQL 쿼리에 필수 보안 필터(테넌트 ID, 민감도 레이블, 사용자 범위)가 애플리케이션 코드가 아닌 데이터베이스 엔진 수준에서 적용되는지 확인하십시오.                 |  1  | D/V |
| 5.4.2 | 모든 벡터 데이터베이스, 검색 인덱스 및 학습 데이터셋에 대해 정책 상속이 적용된 행 수준 보안(RLS) 정책 및 필드 수준 마스킹이 활성화되어 있는지 확인하십시오.                         |  1  | D/V |
| 5.4.3 | 실패한 권한 평가가 쿼리를 즉시 중단하고 빈 결과 집합을 반환하는 대신 명시적인 권한 오류 코드를 반환함으로써 "혼란스러운 대리인 공격(confused deputy attacks)"을 방지하는지 확인하십시오. |  2  |  D  |
| 5.4.4 | 권한 우회가 발생할 수 있는 타임아웃 조건에 대해 자동 알림과 함께 정책 평가 지연 시간을 지속적으로 모니터링하는지 확인하십시오.                                             |  2  |  V  |
| 5.4.5 | 쿼리 재시도 메커니즘이 활성 사용자 세션 내에서 동적 권한 변경을 반영하기 위해 권한 정책을 재평가하는지 확인하십시오.                                                   |  3  | D/V |

## C5.5 출력 필터링 및 데이터 손실 방지

AI 생성 콘텐츠에서 무단 데이터 노출을 방지하기 위해 후처리 제어를 배포합니다.

|   #   | 설명                                                                                    | 레벨  | 역할  |
| :---: | ------------------------------------------------------------------------------------- | :-: | :-: |
| 5.5.1 | 추론 후 필터링 메커니즘이 요청자에게 콘텐츠를 전달하기 전에 무단 PII, 기밀 정보 및 독점 데이터를 스캔하고 삭제하는지 확인하십시오.          |  1  | D/V |
| 5.5.2 | 모델 출력물의 인용, 참고문헌 및 출처 표기가 호출자 권한과 대조되어 유효함을 확인하고, 무단 액세스가 감지될 경우 이를 제거하십시오.           |  1  | D/V |
| 5.5.3 | 사용자 권한 수준 및 데이터 분류에 따라 출력 형식 제한(정제된 PDF, 메타데이터가 제거된 이미지, 승인된 파일 유형)이 제대로 적용되는지 확인합니다. |  2  |  D  |
| 5.5.4 | 편집 알고리즘이 결정론적이며, 버전 관리가 되고, 규정 준수 조사 및 포렌식 분석을 지원하기 위해 감사 로그를 유지하는지 확인하십시오.           |  2  |  V  |
| 5.5.5 | 고위험 편집 이벤트가 데이터 노출 없이 포렌식 복원을 위해 원본 콘텐츠의 암호화 해시를 포함하는 적응형 로그를 생성하는지 확인하십시오.           |  3  |  V  |

## C5.6 다중 테넌트 격리

공유 AI 인프라 내에서 테넌트 간의 암호화 및 논리적 격리를 보장하십시오.

|   #   | 설명                                                                                          | 레벨  | 역할  |
| :---: | ------------------------------------------------------------------------------------------- | :-: | :-: |
| 5.6.1 | 메모리 공간, 임베딩 저장소, 캐시 항목 및 임시 파일이 각 테넌트별로 네임스페이스 분리되어 있으며, 테넌트 삭제나 세션 종료 시 안전하게 삭제되는지 확인하십시오. |  1  | D/V |
| 5.6.2 | 모든 API 요청에 대해 세션 컨텍스트 및 사용자 권한과 암호학적으로 검증된 인증된 테넌트 식별자가 포함되어 있는지 확인하십시오.                    |  1  | D/V |
| 5.6.3 | 서비스 메시 및 컨테이너 오케스트레이션 플랫폼 내에서 테넌트 간 통신에 대해 네트워크 정책이 기본 거부 규칙을 구현하는지 확인하십시오.                 |  2  |  D  |
| 5.6.4 | 고객 관리 키(CMK) 지원 및 테넌트 데이터 저장소 간의 암호학적 격리를 통해 테넌트별로 암호화 키가 고유한지 확인합니다.                       |  3  |  D  |

## C5.7 자율 에이전트 권한 부여

범위가 지정된 권한 토큰과 지속적인 인증을 통해 AI 에이전트 및 자율 시스템에 대한 제어 권한을 관리합니다.

|   #   | 설명                                                                                                      | 레벨  | 역할  |
| :---: | ------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 5.7.1 | 자율 에이전트가 명시적으로 허용된 작업, 접근 가능한 리소스, 시간 경계 및 운영 제약 조건을 열거한 범위 지정 권한 토큰을 받았는지 확인하십시오.                      |  1  | D/V |
| 5.7.2 | 고위험 기능(파일 시스템 접근, 코드 실행, 외부 API 호출, 금융 거래)이 기본적으로 비활성화되어 있으며, 비즈니스 타당성을 명시한 명시적 승인이 있어야만 활성화되는지 확인하십시오. |  1  | D/V |
| 5.7.3 | 기능 토큰이 사용자 세션에 바인딩되어 있고, 암호화 무결성 보호가 포함되어 있으며, 오프라인 시나리오에서 토큰이 저장되거나 재사용될 수 없도록 확인하십시오.                 |  2  |  D  |
| 5.7.4 | 에이전트가 시작한 작업이 전체 컨텍스트 평가와 감사 로그를 포함한 ABAC 정책 엔진을 통한 2차 승인 절차를 거치는지 확인합니다.                               |  2  |  V  |
| 5.7.5 | 에이전트 오류 조건 및 예외 처리가 사고 분석 및 법의학 조사를 지원하기 위해 능력 범위 정보를 포함하는지 확인하십시오.                                     |  3  |  V  |
