# C4 인프라스트럭처, 구성 및 배포 보안

## 제어 목표

AI 인프라는 권한 상승, 공급망 변조, 횡적 이동에 대해 보안 설정, 실행 시 격리, 신뢰할 수 있는 배포 파이프라인, 포괄적 모니터링을 통해 견고하게 보호되어야 합니다. 검증되고 권한이 부여된 인프라 구성 요소만 보안, 무결성, 감사 가능성을 보장하는 통제된 프로세스를 거쳐 운영 환경에 배포됩니다.

핵심 보안 목표: 암호화 서명되고 취약점 검사를 거친 인프라 구성 요소만 보안 정책을 시행하고 변경 불가능한 감사 추적을 유지하는 자동화된 검증 파이프라인을 통해 운영 환경에 도달합니다.

---

## C4.1 런타임 환경 격리

OS 수준의 격리 원시 기능을 통해 컨테이너 탈출 및 권한 상승을 방지합니다.

|   #   | 설명                                                                                                      | 수준  | 역할  |
| :---: | ------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 4.1.1 | 예를 들어 컨테이너의 불필요한 Linux 권한을 제거하여 모든 AI 작업이 운영 체제에서 최소한의 권한으로 실행되는지 확인합니다.                                |  1  | D/V |
| 4.1.2 | 워크로드가 샌드박싱, seccomp 프로파일, AppArmor, SELinux 또는 이와 유사한 악용 방지 기술에 의해 보호되고 있으며, 구성도 적절한지 검증하십시오.           |  1  | D/V |
| 4.1.3 | 워크로드가 읽기 전용 루트 파일 시스템에서 실행되며, 쓰기 가능한 마운트가 명시적으로 정의되고 제한적인 옵션(예: noexec, nosuid, nodev)으로 강화되었는지 확인하십시오. |  2  | D/V |
| 4.1.4 | 런타임 모니터링이 권한 상승 및 컨테이너 탈출 동작을 감지하고, 위반 프로세스를 자동으로 종료하는지 확인합니다.                                          |  2  | D/V |
| 4.1.5 | 고위험 AI 작업 부하는 원격 증명이 성공적으로 완료된 후에만 하드웨어 격리 환경(예: TEE, 신뢰할 수 있는 하이퍼바이저 또는 베어메탈 노드)에서 실행되는지 검증하십시오.       |  3  | D/V |

## C4.2 보안 빌드 및 배포 파이프라인

재현 가능한 빌드와 서명된 아티팩트를 통해 암호화 무결성과 공급망 보안을 보장하십시오.

|   #   | 설명                                                                             | 수준  | 역할  |
| :---: | ------------------------------------------------------------------------------ | :-: | :-: |
| 4.2.1 | 빌드가 재현 가능하며 빌드 산출물에 대해 독립적으로 검증할 수 있는 서명된 출처 메타데이터를 적절히 생성하는지 확인하십시오.          |  1  | D/V |
| 4.2.2 | 빌드가 소프트웨어 자재 명세서(SBOM)를 생성하고 배포를 위해 수락되기 전에 서명되었는지 확인하세요.                      |  2  | D/V |
| 4.2.3 | 배포 시 빌드 산출물(예: 컨테이너 이미지)의 서명 및 출처 메타데이터가 검증되었는지 확인하고, 검증되지 않은 산출물은 거부되도록 하십시오. |  2  | D/V |

## C4.3 네트워크 보안 및 접근 제어

기본 거부 정책과 암호화된 통신을 사용하여 제로 트러스트 네트워킹을 구현하십시오.

|   #   | 설명                                                                                                                       | 수준  | 역할  |
| :---: | ------------------------------------------------------------------------------------------------------------------------ | :-: | :-: |
| 4.3.1 | 네트워크 정책이 기본적으로 수신 및 송신을 차단하며, 필요한 서비스만 명시적으로 허용하는지 확인하십시오.                                                               |  1  | D/V |
| 4.3.2 | 관리자 액세스 프로토콜(예: SSH, RDP) 및 클라우드 메타데이터 서비스에 대한 액세스가 제한되며 강력한 인증을 요구하는지 확인하십시오.                                           |  1  | D/V |
| 4.3.3 | 출력 트래픽이 승인된 목적지로 제한되고 모든 요청이 기록되는지 확인하십시오.                                                                               |  2  | D/V |
| 4.3.4 | 서비스 간 통신이 상호 TLS를 사용하며, 인증서 검증과 정기적인 자동 교체가 이루어지는지 확인하십시오.                                                               |  2  | D/V |
| 4.3.5 | AI 워크로드 및 환경(dev, test, prod)이 직접적인 인터넷 액세스 없이 격리된 네트워크 세그먼트(VPC/VNet)에서 실행되며, 공유된 IAM 역할, 보안 그룹 또는 환경 간 연결이 없음을 확인하십시오. |  2  | D/V |

## C4.4 비밀 및 암호 키 관리

보안 저장소, 자동 회전, 강력한 접근 제어로 비밀과 암호화 키를 보호하세요.

|   #   | 설명                                                                                                                | 수준  | 역할  |
| :---: | ----------------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 4.4.1 | 비밀은 전용 비밀 관리 시스템에 저장되어야 하며, 저장 시 암호화되어야 하고 애플리케이션 작업 부하와 분리되어 있는지 확인하십시오.                                         |  1  | D/V |
| 4.4.2 | 암호화 키가 하드웨어 기반 모듈(예: HSM, 클라우드 KMS)에서 생성되고 저장되는지 검증하십시오.                                                          |  1  | D/V |
| 4.4.3 | 프로덕션 비밀에 대한 접근이 강력한 인증을 요구하는지 확인하십시오.                                                                             |  1  | D/V |
| 4.4.4 | 비밀은 전용 비밀 관리 시스템을 통해 런타임에 애플리케이션에 배포되는지 확인해야 합니다. 비밀은 절대로 소스 코드, 구성 파일, 빌드 아티팩트, 컨테이너 이미지 또는 환경 변수에 포함되어서는 안 됩니다. |  1  | D/V |
| 4.4.5 | 비밀 값 회전이 자동화되었는지 확인합니다.                                                                                           |  2  | D/V |

## C4.5 AI 작업 부하 샌드박싱 및 검증

신뢰할 수 없는 AI 모델을 보안 샌드박스에서 격리하고, 신뢰할 수 있는 실행 환경(TEE)과 기밀 컴퓨팅 기술을 사용하여 민감한 AI 작업을 보호합니다.

|   #   | 설명                                                                                | 수준  | 역할  |
| :---: | --------------------------------------------------------------------------------- | :-: | :-: |
| 4.5.1 | 외부 또는 신뢰할 수 없는 AI 모델이 격리된 샌드박스에서 실행되는지 확인하십시오.                                    |  1  | D/V |
| 4.5.2 | 샌드박스 작업이 기본적으로 외부 네트워크 연결이 없으며, 필요한 접근 권한이 명시적으로 정의되어 있는지 확인하십시오.                 |  1  | D/V |
| 4.5.3 | 모델 또는 작업 부하 로딩 전에 작업 부하 증명이 수행되어 신뢰할 수 있는 실행 환경에 대한 암호화된 증명이 보장되는지 확인하십시오.        |  2  | D/V |
| 4.5.4 | 비밀 워크로드가 하드웨어 기반 격리, 메모리 암호화 및 무결성 보호를 제공하는 신뢰할 수 있는 실행 환경(TEE) 내에서 실행되는지 확인하십시오. |  3  | D/V |
| 4.5.5 | 암호화된 연산과 봉인된 모델 가중치 및 보호된 실행을 통해 모델 추출을 방지하는 기밀 추론 서비스임을 확인하십시오.                  |  3  | D/V |
| 4.5.6 | 신뢰할 수 있는 실행 환경의 오케스트레이션에는 수명 주기 관리, 원격 증명, 그리고 암호화된 통신 채널이 포함되는지 확인하십시오.          |  3  | D/V |
| 4.5.7 | 보안 다자간 계산(SMPC)이 개별 데이터셋이나 모델 매개변수를 노출하지 않고 협업 AI 훈련을 가능하게 하는지 검증하십시오.            |  3  | D/V |

## C4.6 AI 인프라 자원 관리, 백업 및 복구

쿼터 및 모니터링을 통해 자원 소모 공격을 방지하고 공정한 자원 할당을 보장합니다. 안전한 백업, 검증된 복구 절차 및 재해 복구 기능을 통해 인프라 복원력을 유지합니다.

|   #   | 설명                                                                                                                           | 수준  | 역할  |
| :---: | ---------------------------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 4.6.1 | 서비스 거부 공격을 완화하기 위해 Kubernetes ResourceQuotas 또는 유사한 수단으로 워크로드의 리소스 소비가 적절하게 제한되었는지 확인하십시오.                                   |  2  | D/V |
| 4.6.2 | 정의된 CPU, 메모리 또는 요청 임계값을 초과할 경우 리소스 소진이 자동 보호 기능(예: 속도 제한 또는 작업 부하 격리)을 유발하는지 확인합니다.                                          |  2  | D/V |
| 4.6.3 | 백업 시스템이 별도의 자격 증명을 가진 분리된 네트워크에서 운영되는지 확인하고, 저장 시스템이 에어갭 네트워크에서 운영되거나 무단 변경을 방지하는 WORM(쓰기 한 번, 여러 번 읽기) 보호 기능을 구현하는지 검증하십시오. |  2  | D/V |

## C4.7 AI 하드웨어 보안

GPU, TPU 및 전문 AI 가속기를 포함한 AI 전용 하드웨어 구성 요소를 보호하십시오.

|   #   | 설명                                                                                                               | 수준  | 역할  |
| :---: | ---------------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 4.7.1 | 작업 부하 실행 전에 하드웨어 기반 증명 메커니즘(예: TPM, DRTM 또는 동등한 기술)을 사용하여 AI 가속기의 무결성이 검증되었는지 확인하십시오.                            |  2  | D/V |
| 4.7.2 | 가속기(GPU) 메모리가 작업 간 분할 메커니즘을 통해 분리되고 작업 간 메모리 초기화가 이루어지는지 검증하십시오.                                                 |  2  | D/V |
| 4.7.3 | 하드웨어 보안 모듈(HSM)이 AI 모델 가중치와 암호화 키를 FIPS 140-3 Level 3 또는 Common Criteria EAL4+ 인증과 함께 보호하는지 확인하십시오.              |  3  | D/V |
| 4.7.4 | 가속기 펌웨어(GPU/TPU/NPUs)가 버전 고정되어 있으며, 부팅 시 서명되고 인증되었는지 확인합니다; 서명되지 않았거나 디버그용 펌웨어는 차단됩니다.                           |  2  | D/V |
| 4.7.5 | 작업/테넌트 간에 VRAM 및 온패키지 메모리가 초기화되었는지 확인하고, 장치 재설정 정책이 테넌트 간 데이터 잔류를 방지하는지 검증하십시오.                                  |  2  | D/V |
| 4.7.6 | 파티셔닝/격리 기능(예: MIG/VM 파티셔닝)이 테넌트별로 적용되어 파티션 간 피어 투 피어 메모리 접근을 차단하는지 확인하십시오.                                       |  2  | D/V |
| 4.7.7 | 가속기 인터커넥트(NVLink/PCIe/InfiniBand/RDMA/NCCL)가 승인된 토폴로지 및 인증된 엔드포인트로 제한되어 있는지 확인하고, 평문 교차 테넌트 링크는 허용되지 않도록 해야 합니다. |  3  | D/V |
| 4.7.8 | 가속기 텔레메트리(전력, 온도, ECC, 성능 카운터)가 SIEM/OTel로 내보내지고, 사이드 채널 또는 은밀 채널을 나타내는 이상 징후에 대해 경고가 발생하는지 확인하십시오.              |  3  |  D  |

## C4.8 에지 및 분산 AI 보안

에지 컴퓨팅, 연합 학습 및 다중 사이트 아키텍처를 포함한 보안 분산 AI 배포.

|   #   | 설명                                                                        | 수준  | 역할  |
| :---: | ------------------------------------------------------------------------- | :-: | :-: |
| 4.8.1 | 에지 AI 장치가 상호 TLS를 사용하여 중앙 인프라에 인증하는지 확인하십시오.                              |  2  | D/V |
| 4.8.2 | 에지 디바이스가 검증된 서명 및 롤백 보호 기능을 통해 보안 부팅을 구현하여 펌웨어 다운그레이드 공격을 방지하는지 확인하십시오.   |  2  | D/V |
| 4.8.3 | 분산 AI 조정이 참가자 검증 및 악성 노드 탐지를 포함한 비잔틴 장애 허용 합의 메커니즘을 사용하는지 확인하십시오.         |  3  | D/V |
| 4.8.4 | 엣지-투-클라우드 통신이 대역폭 제한, 데이터 압축, 암호화된 로컬 스토리지를 이용한 보안 오프라인 작동을 지원하는지 확인하십시오. |  3  | D/V |
