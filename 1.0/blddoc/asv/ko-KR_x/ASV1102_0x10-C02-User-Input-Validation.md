# C2 사용자 입력 검증

## 제어 목표

사용자 입력에 대한 견고한 검증은 AI 시스템에 대한 가장 심각한 공격 중 일부에 대한 1차 방어선입니다. 프롬프트 인젝션 공격은 시스템 지침을 무시하거나 민감한 데이터를 유출시키거나 모델을 허용되지 않은 행동으로 유도할 수 있습니다. 전용 필터 및 기타 검증 수단이 마련되지 않는 한, 연구에 따르면 문맥 창을 악용한 탈옥 공격은 계속해서 효과적일 것입니다.

---

## C2.1 프롬프트 인젝션 방어

프롬프트 인젝션은 AI 시스템에서 가장 큰 위험 중 하나입니다. 이에 대한 방어는 패턴 필터, 데이터 분류기 및 명령 계층 구조 강화를 조합하여 수행됩니다.

|   #   | 설명                                                                                                                                                                                                                                               | 수준  | 역할  |
| :---: | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | :-: | :-: |
| 2.1.1 | 행동을 유도할 수 있는 외부 또는 파생 입력(사용자 프롬프트, RAG 결과, 플러그인 또는 MCP 출력, 에이전트 간 메시지, API 또는 웹후크 응답, 구성 또는 정책 파일, 메모리 읽기 및 쓰기 포함)을 신뢰할 수 없는 것으로 취급하고, 인용 또는 태깅 및 활성 콘텐츠 제거를 통해 비활성화하며, 프롬프트에 연결하거나 작업을 실행하기 전에 유지 관리되는 프롬프트 인젝션 탐지 규칙 집합 또는 서비스로 검토되었는지 확인하십시오. |  1  | D/V |
| 2.1.2 | 시스템이 사용자 지침 처리 이후에도 시스템 및 개발자 메시지가 사용자 지침 및 기타 신뢰할 수 없는 입력을 무시하고 우선하는 명령 계층 구조를 적용하는지 확인하십시오.                                                                                                                                                    |  1  | D/V |
| 2.1.4 | 웹 페이지, PDF, 이메일과 같은 제3자 콘텐츠에서 유래된 프롬프트가 메인 프롬프트에 결합되기 전에 별도로 정제되는지 확인하십시오(예: 명령어 유사 지시문 제거 및 HTML, Markdown, 스크립트 콘텐츠 중성화).                                                                                                                      |  2  |  D  |

## C2.2 적대적 예제 저항성

자연어 처리(NLP) 모델은 인간이 종종 놓치는 미묘한 문자 또는 단어 수준의 교란에 여전히 취약하며, 모델은 이를 오분류하는 경향이 있습니다.

|   #   | 설명                                                                                                                                                    | 수준  | 역할  |
| :---: | ----------------------------------------------------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 2.2.1 | 기본 입력 정규화 단계(Unicode NFC, 동형문자 매핑, 공백 다듬기, 제어 문자 및 보이지 않는 유니코드 문자 제거)가 토큰화 또는 임베딩 전에 그리고 툴이나 MCP 인수로 구문 분석하기 전에 실행되는지 확인하십시오.                         |  1  |  D  |
| 2.2.2 | 통계적 이상 탐지가 언어 규범과 비정상적으로 높은 편집 거리 또는 비정상적인 임베딩 거리를 가진 입력을 표시하는지 확인하고, 표시된 입력이 프롬프트에 연결되거나 작업이 실행되기 전에 차단되는지 확인하십시오.                                   |  2  | D/V |
| 2.2.3 | 추론 파이프라인이 고위험 엔드포인트에 대해 적대적 학습으로 강화된 모델 변형이나 방어 계층(예: 무작위화, 방어적 증류, 정렬 검사)을 지원하는지 확인하십시오.                                                             |  2  |  D  |
| 2.2.4 | 의심되는 적대적 입력이 격리되고 전체 페이로드 및 추적 메타데이터(출처, 도구 또는 MCP 서버, 에이전트 ID, 세션)와 함께 기록되었는지 확인합니다.                                                                 |  2  |  V  |
| 2.2.5 | 입력 및 출력 모두에서 인코딩 및 표현 스머글링(예: 보이지 않는 유니코드/제어 문자, 동형 문자 교체, 또는 혼합 방향 텍스트)이 감지되고 완화되었는지 확인합니다. 승인된 완화 방법에는 정규화, 엄격한 스키마 검증, 정책 기반 거부, 또는 명시적 표시가 포함됩니다. |  2  | D/V |

## C2.3 프롬프트 문자 집합

비즈니스 요구 사항에 필요한 문자만 허용하도록 사용자 입력의 문자 집합을 제한하는 것은 다양한 유형의 공격을 방지하는 데 도움이 될 수 있습니다.

|   #   | 설명                                                                                    | 수준  | 역할  |
| :---: | ------------------------------------------------------------------------------------- | :-: | :-: |
| 2.3.1 | 시스템이 사용자 입력에 대해 문자 집합 제한을 구현하여 비즈니스 목적상 명시적으로 필요한 문자만 허용하는지 확인하십시오.                   |  1  |  D  |
| 2.3.2 | 허용된 문자 집합을 정의하기 위해 허용 목록 방식을 사용했는지 확인하십시오.                                            |  1  |  D  |
| 2.3.3 | 허용된 집합 외의 문자를 포함하는 입력이 거부되고 추적 메타데이터(출처, 도구 또는 MCP 서버, 에이전트 ID, 세션)와 함께 기록되는지 확인하십시오. |  1  | D/V |

## C2.4 스키마, 타입 및 길이 검증

잘못된 형식이거나 과도하게 큰 입력을 포함한 AI 공격은 구문 분석 오류, 필드 간 프롬프트 유출, 자원 고갈을 초래할 수 있습니다. 엄격한 스키마 적용은 결정론적 도구 호출을 수행할 때에도 필수 조건입니다.

|   #   | 설명                                                                                                                                                          | 수준  | 역할  |
| :---: | ----------------------------------------------------------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 2.4.1 | 모든 API, 도구 또는 MCP 엔드포인트가 명확한 입력 스키마(JSON 스키마, Protobuf 또는 다중 모달 동등체)를 정의하고, 추가적이거나 알 수 없는 필드와 암묵적 유형 강제를 거부하며, 프롬프트 조립 또는 도구 실행 전에 서버 측에서 입력을 검증하는지 확인하십시오. |  1  |  D  |
| 2.4.2 | 입력이 최대 토큰 또는 바이트 한도를 초과할 경우 안전한 오류로 거부되며 절대 조용히 잘리지 않는지 확인하십시오.                                                                                             |  1  | D/V |
| 2.4.3 | 도구 또는 MCP 인수에 대해서도 서버 측에서 유형 검사(예: 숫자 범위, 열거형 값, 이미지/오디오 MIME 유형)가 적용되는지 확인하십시오.                                                                            |  2  | D/V |
| 2.4.4 | NLP 입력을 이해하는 의미 검증기가 일정한 시간 내에 실행되며 알고리즘적 DoS를 방지하기 위해 외부 네트워크 호출을 하지 않는지 확인하십시오.                                                                           |  2  |  D  |
| 2.4.5 | 검증 실패가 레드액션된 페이로드 스니펫과 명확한 오류 코드로 기록되며 보안 분류 지원을 위해 추적 메타데이터(출처, 도구 또는 MCP 서버, 에이전트 ID, 세션)를 포함하는지 확인하십시오.                                                  |  3  |  V  |

## C2.5 콘텐츠 및 정책 심사

개발자는 허용되지 않은 콘텐츠(예: 불법 지침, 혐오 발언 및/또는 저작권이 있는 텍스트)를 요청하는 구문적으로 유효한 프롬프트를 감지할 수 있어야 하며, 이를 확산하지 않도록 차단해야 합니다.

|   #   | 설명                                                                                                                                             | 수준  | 역할  |
| :---: | ---------------------------------------------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 2.5.1 | 콘텐츠 분류기(제로샷 또는 파인튜닝된)가 모든 입력과 출력을 폭력, 자해, 증오, 성적 콘텐츠 및 불법 요청에 대해 점수화하며, 구성 가능한 임계값을 갖추었는지 확인하십시오.                                              |  1  |  D  |
| 2.5.2 | 정책을 위반하는 입력이 거부되어 하류 LLM 또는 도구/MCP 호출로 전파되지 않도록 확인하십시오.                                                                                        |  1  | D/V |
| 2.5.3 | 요청 시점에 해결되는 에이전트 역할 속성을 포함한 속성 기반 규칙을 통해 사용자별 정책(연령, 지역 법적 제약)을 준수하는지 검증합니다.                                                                   |  2  |  D  |
| 2.5.4 | 스크리닝 로그에 분류기 신뢰도 점수와 정책 카테고리 태그가 적용 단계(프리-프롬프트 또는 포스트-응답) 및 추적 메타데이터(출처, 도구 또는 MCP 서버, 에이전트 ID, 세션)와 함께 포함되어 SOC 상관관계 및 향후 레드-팀 재현을 위해 확인하십시오. |  3  |  V  |

| 2.7.2 | 파일이 수집되기 전에 악성 코드 및 스테가노그래피 페이로드에 대해 스캔되는지 확인하고, 모든 활성 콘텐츠(스크립트나 매크로와 같은)가 제거되거나 파일이 격리되는지 검증합니다.                                                                             |  2  | D/V |

| 2.7.4 | 멀티모달 입력 검증 실패가 기록되고 추적 메타데이터(출처, 도구 또는 MCP 서버, 에이전트 ID, 세션)와 함께 조사 경고를 발생시키는지 확인하십시오.                                                                                         |  3  |  V  |
| 2.7.5 | 교차 모달 공격 탐지가 상관 관계 규칙과 경고 생성을 통해 여러 입력 유형에 걸친 협력 공격(예: 이미지 내 스테가노그래피 페이로드와 텍스트 내 프롬프트 인젝션 결합)을 식별하는지, 그리고 확인된 탐지가 차단되거나 HITL(사람 개입) 승인을 필요로 하는지 검증합니다.                        |  2  | D/V |

| 2.8.3 | 적응형 탐지 모델이 최근 공격 활동에 따라 민감도를 조정하고, 새로운 패턴을 실시간으로 업데이트하며, 위험 적응형 대응(예: 도구 비활성화, 컨텍스트 축소, 또는 HITL 승인 요구)을 유발하는지 확인하십시오. |  2  | D/V |
| 2.8.4 | 사용자 히스토리, 출처 및 세션 행동(출처, 도구 또는 MCP 서버, 에이전트 ID, 세션 포함)과 같은 트레이스 메타데이터의 맥락 분석을 통해 탐지 정확도가 향상되었는지 확인하십시오.               |  3  | D/V |
| 2.8.5 | 탐지 성능 지표(탐지율, 오탐률, 처리 지연 시간)가 지속적으로 모니터링되고 최적화되는지 확인하며, 여기에는 차단 시간 및 단계(사전 프롬프트/응답 후)가 포함됩니다.                         |  3  | D/V |
