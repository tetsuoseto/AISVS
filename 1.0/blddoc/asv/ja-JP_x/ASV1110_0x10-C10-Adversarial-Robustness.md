# 10 敵対的ロバストネスとプライバシー防御

## 制御目標

AIモデルが回避、推論、抽出、または毒物攻撃に直面した際にも、信頼性が高く、プライバシーを保護し、不正使用に強い状態を維持することを保証します。

---

## 10.1 モデルの整合性と安全性

有害またはポリシー違反の出力を防止する。

|   #    | 説明                                                                                   | レベル | 役割  |
| :----: | ------------------------------------------------------------------------------------ | :-: | :-: |
| 10.1.1 | アライメントテストスイート（レッドチームプロンプト、脱獄プローブ、禁止コンテンツ）がバージョン管理され、すべてのモデルリリース時に実行されていることを確認してください。 |  1  | D/V |
| 10.1.2 | 拒否および安全完了のガードレールが適用されていることを確認してください。                                                 |  1  |  D  |
| 10.1.3 | 自動評価者が有害コンテンツ率を測定し、設定された閾値を超えるリグレッションを検出することを確認してください。                               |  2  | D/V |
| 10.1.4 | カウンタージェイルブレイクトレーニングが文書化され、再現可能であることを検証してください。                                        |  2  |  D  |
| 10.1.5 | 正式なポリシー遵守証明または認定された監視が重要な領域をカバーしていることを検証してください。                                      |  3  |  V  |

---

## 10.2 敵対的例の堅牢化

操作された入力に対する回復力を向上させる。堅牢な敵対的トレーニングとベンチマークスコアリングが現在の最良の実践である。

|   #    | 説明                                                      | レベル | 役割  |
| :----: | ------------------------------------------------------- | :-: | :-: |
| 10.2.1 | プロジェクトのリポジトリに再現可能なシードを用いた敵対的トレーニング構成が含まれていることを確認してください。 |  1  |  D  |
| 10.2.2 | 対抗的サンプル検出が本番パイプラインでブロッキングアラートを発生させることを検証する。             |  2  | D/V |
| 10.2.4 | 認定堅牢性の証明または区間境界証明書が、少なくとも最も重要な主要クラスをカバーしていることを確認してください。 |  3  |  V  |
| 10.2.5 | 回帰テストが適応的攻撃を使用して、測定可能なロバストネスの低下がないことを確認すること。            |  3  |  V  |

---

## 10.3 メンバーシップ推論緩和

レコードがトレーニングデータに含まれていたかどうかを判断する能力を制限する。差分プライバシーと信頼度スコアのマスキングが、現在知られている最も効果的な防御策であり続けている。

|   #    | 説明                                                                      | レベル | 役割  |
| :----: | ----------------------------------------------------------------------- | :-: | :-: |
| 10.3.1 | クエリごとのエントロピー正則化または温度スケーリングが過剰に確信した予測を減少させることを検証する。                      |  1  |  D  |
| 10.3.2 | 機微なデータセットに対しては、ε境界付き差分プライバシー最適化を用いてトレーニングが行われていることを検証してください。            |  2  |  D  |
| 10.3.3 | 攻撃シミュレーション（シャドウモデルまたはブラックボックス）が、保持されたデータに対して攻撃AUC ≤ 0.60を示すことを確認してください。 |  2  |  V  |

---

## 10.4 モデル逆転耐性

プライベート属性の再構成を防ぐ。最近の調査では、実用的な防御策として出力の切り捨てとDP保証が強調されている。

|   #    | 説明                                                           | レベル | 役割  |
| :----: | ------------------------------------------------------------ | :-: | :-: |
| 10.4.1 | 機密属性が直接出力されることが決してないことを確認してください。必要に応じて、バケット化や一方向変換を使用してください。 |  1  |  D  |
| 10.4.2 | 同じプリンシパルからの繰り返しの適応クエリがクエリレート制限によって制御されていることを確認してください。        |  1  | D/V |
| 10.4.3 | モデルがプライバシー保護のためのノイズを用いて訓練されていることを確認してください。                   |  2  |  D  |

---

## 10.5 モデル抽出防御

不正なクローンを検出し抑止します。ウォーターマーキングとクエリパターン分析が推奨されます。

|   #    | 説明                                                                    | レベル | 役割  |
| :----: | --------------------------------------------------------------------- | :-: | :-: |
| 10.5.1 | 推論ゲートウェイが、モデルの記憶閾値に合わせて調整されたグローバルおよびAPIキーごとのレート制限を適用していることを確認してください。  |  1  |  D  |
| 10.5.2 | クエリエントロピーと入力複数性の統計が自動抽出検出器にフィードされることを検証してください。                        |  2  | D/V |
| 10.5.3 | 壊れやすいまたは確率的なウォーターマークが、疑わしいクローンに対して1000回以下のクエリでp < 0.01の証明ができることを検証する。 |  2  |  V  |
| 10.5.4 | ウォーターマークキーとトリガーセットがハードウェアセキュリティモジュールに保存され、毎年ローテーションされていることを確認してください。  |  3  |  D  |
| 10.5.5 | 抽出アラートイベントに問題のあるクエリが含まれており、インシデント対応プレイブックと統合されていることを確認してください。         |  3  |  V  |

---

## 10.6 推論時の毒データ検出

バックドアや毒された入力を特定し、無効化する。

|   #    | 説明                                                              | レベル | 役割  |
| :----: | --------------------------------------------------------------- | :-: | :-: |
| 10.6.1 | モデル推論の前に、入力が異常検知器（例：STRIP、整合性スコアリング）を通過することを検証してください。           |  1  |  D  |
| 10.6.2 | 検出器の閾値が、クリーンおよび汚染された検証セットで調整され、誤検出率が5%未満になることを確認してください。         |  1  |  V  |
| 10.6.3 | 毒性があるとフラグ付けされた入力がソフトブロッキングおよび人間によるレビューのワークフローを引き起こすことを確認してください。 |  2  |  D  |
| 10.6.4 | 検出器が適応型トリガーレスバックドア攻撃によってストレステストされていることを検証する。                    |  2  |  V  |
| 10.6.5 | 検出効果の指標が記録され、新しい脅威インテリジェンスに基づいて定期的に再評価されていることを確認してください。         |  3  |  D  |

## 10.7 動的セキュリティポリシー適応

脅威インテリジェンスと行動分析に基づくリアルタイムのセキュリティポリシー更新。

|   #    | 説明                                                                | レベル | 役割  |
| :----: | ----------------------------------------------------------------- | :-: | :-: |
| 10.7.1 | エージェントの再起動なしでセキュリティポリシーを動的に更新でき、かつポリシーバージョンの整合性が維持されることを検証してください。 |  1  | D/V |
| 10.7.2 | ポリシーの更新が認可されたセキュリティ担当者によって暗号的に署名され、適用前に検証されていることを確認してください。        |  2  | D/V |
| 10.7.3 | 動的ポリシー変更が、正当化、承認チェーン、およびロールバック手順を含む完全な監査記録と共に記録されていることを確認してください。  |  2  | D/V |
| 10.7.4 | 適応型セキュリティメカニズムがリスクコンテキストおよび行動パターンに基づいて脅威検出の感度を調整することを確認してください。    |  3  | D/V |
| 10.7.5 | ポリシー適応の決定が説明可能であり、セキュリティチームのレビューのための証拠の記録を含んでいることを確認してください。       |  3  | D/V |

## 10.8 リフレクションベースのセキュリティ分析

エージェントの自己反省とメタ認知分析によるセキュリティ検証。

|   #    | 説明                                                                 | レベル | 役割  |
| :----: | ------------------------------------------------------------------ | :-: | :-: |
| 10.8.1 | エージェントの反省メカニズムに、意思決定および行動に対するセキュリティ重視の自己評価が含まれていることを確認してください。      |  1  | D/V |
| 10.8.2 | 反射出力が検証され、敵対的入力による自己評価メカニズムの操作を防止していることを確認してください。                  |  2  | D/V |
| 10.8.3 | メタ認知的セキュリティ分析が、エージェントの推論プロセスにおける潜在的なバイアス、操作、または妥協を特定することを確認してください。 |  2  | D/V |
| 10.8.4 | リフレクションベースのセキュリティ警告が強化された監視および潜在的な人的介入のワークフローをトリガーすることを検証してください。   |  3  | D/V |
| 10.8.5 | セキュリティ反省からの継続的な学習が正当な機能を損なうことなく脅威検出を向上させることを検証する。                  |  3  | D/V |

## 10.9 進化と自己改善のセキュリティ

自己修正および進化が可能なエージェントシステムのためのセキュリティ制御。

|   #    | 説明                                                 | レベル | 役割  |
| :----: | -------------------------------------------------- | :-: | :-: |
| 10.9.1 | 自己修正機能が指定された安全領域内に制限され、正式な検証境界が設けられていることを確認してください。 |  1  | D/V |
| 10.9.2 | 進化提案が実装前にセキュリティ影響評価を受けることを確認してください。                |  2  | D/V |
| 10.9.3 | 自己改善メカニズムが整合性検証を伴うロールバック機能を含んでいることを確認してください。       |  2  | D/V |
| 10.9.4 | メタラーニングのセキュリティが改善アルゴリズムの敵対的操作を防止することを検証してください。     |  3  | D/V |
| 10.9.5 | 再帰的な自己改善が収束の数学的証明を伴う形式的な安全制約によって制限されていることを検証する。    |  3  | D/V |
