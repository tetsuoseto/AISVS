# C2 ユーザー入力検証

## 制御目標

ユーザー入力の堅牢な検証は、AIシステムに対する最も破壊的な攻撃のいくつかに対する第一の防御線です。プロンプトインジェクション攻撃は、システムの指示を上書きしたり、機密データを漏えいさせたり、モデルを許可されていない行動に誘導したりすることがあります。専用のフィルターやその他の検証がない限り、研究によればコンテキストウィンドウを悪用したジャイルブレイクは引き続き有効であることが示されています。

---

## C2.1 プロンプトインジェクション防御

プロンプトインジェクションは、AIシステムにおける主要なリスクの一つです。この戦術に対する防御は、パターンフィルター、データ分類器、および命令階層の強制を組み合わせて行われます。

|   #   | 説明                                                                                                                                                                                                                                           | レベル | 役割  |
| :---: | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 2.1.1 | ユーザープロンプト、RAG結果、プラグインまたはMCPの出力、エージェント間メッセージ、APIまたはウェブフックの応答、設定またはポリシーファイル、メモリの読み書きなど、動作を誘導する可能性のある外部または派生入力がすべて信頼されていないものとして扱われ、引用やタグ付け、アクティブコンテンツの除去によって無効化され、プロンプトへの連結またはアクションの実行前に、維持されたプロンプトインジェクション検出ルールセットまたはサービスによって検査されることを確認してください。 |  1  | D/V |
| 2.1.2 | システムが指示の階層を強制し、システムメッセージや開発者メッセージがユーザーの指示やその他の信頼できない入力よりも優先されることを、ユーザー指示の処理後も含めて検証してください。                                                                                                                                                    |  1  | D/V |
| 2.1.4 | サードパーティのコンテンツ（ウェブページ、PDF、電子メール）に由来するプロンプトが、主要なプロンプトに連結される前に、指示のような指令を除去し、HTML、Markdown、スクリプト内容を無効化するなどの方法で単独で安全に処理されていることを検証してください。                                                                                                          |  2  |  D  |

---

## C2.2 対抗的例抵抗性

自然言語処理（NLP）モデルは、しばしば人間が見逃す微妙な文字レベルや単語レベルの摂動に対して依然として脆弱であり、モデルはそれらを誤分類しやすい。

|   #   | 説明                                                                                                                                                          | レベル | 役割  |
| :---: | ----------------------------------------------------------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 2.2.1 | 基本的な入力正規化手順（Unicode NFC、同形異字体のマッピング、空白のトリミング、制御文字および不可視Unicode文字の除去）が、トークン化または埋め込みの前、およびツールやMCPの引数への解析の前に実行されていることを確認してください。                                |  1  |  D  |
| 2.2.2 | 統計的異常検出が、言語の規範から異常に高い編集距離や異常な埋め込み距離を持つ入力を検出し、検出された入力がプロンプトへの連結やアクションの実行前にゲートされることを検証してください。                                                                 |  2  | D/V |
| 2.2.3 | 推論パイプラインが、高リスクエンドポイント向けに敵対的訓練で強化されたモデルのバリアントや防御層（例：ランダム化、防御的蒸留、アライメントチェック）をサポートしていることを検証してください。                                                             |  2  |  D  |
| 2.2.4 | 疑わしい敵対的入力が隔離され、完全なペイロードとトレースメタデータ（ソース、ツールまたはMCPサーバー、エージェントID、セッション）と共にログに記録されていることを確認してください。                                                                |  2  |  V  |
| 2.2.5 | 入力および出力の両方におけるエンコーディングおよび表現のスムーフィング（例：不可視のUnicode/制御文字、ホモグリフの置換、混在方向のテキスト）が検出され、軽減されていることを確認してください。承認された軽減手段には、正規化、厳密なスキーマ検証、ポリシーベースの拒否、または明示的なマーキングが含まれます。 |  2  | D/V |

## C2.3 プロンプト文字セット

ユーザー入力の文字セットを業務要件に必要な文字のみに制限することは、さまざまな種類の攻撃を防ぐのに役立ちます。

|   #   | 説明                                                                                     | レベル | 役割  |
| :---: | -------------------------------------------------------------------------------------- | :-: | :-: |
| 2.3.1 | システムがユーザー入力に対して文字セットの制限を実装しており、業務上明示的に必要な文字のみを許可していることを確認してください。                       |  1  |  D  |
| 2.3.2 | 許可された文字セットを定義するために、許可リストアプローチが使用されていることを確認してください。                                      |  1  |  D  |
| 2.3.3 | 許可されたセット外の文字を含む入力が拒否され、トレースメタデータ（ソース、ツールまたはMCPサーバー、エージェントID、セッション）と共に記録されることを確認してください。 |  1  | D/V |

## C2.4 スキーマ、タイプおよび長さの検証

不正な形式や過大な入力を特徴とするAI攻撃は、パースエラー、フィールド間でのプロンプトの漏れ、リソースの枯渇を引き起こす可能性があります。決定的なツール呼び出しを行う際にも、厳密なスキーマの適用が前提条件となります。

|   #   | 説明                                                                                                                                                 | レベル | 役割  |
| :---: | -------------------------------------------------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 2.4.1 | すべてのAPI、ツール、またはMCPエンドポイントが明確な入力スキーマ（JSONスキーマ、Protobuf、またはマルチモーダルの同等物）を定義し、不明なフィールドや追加フィールド、暗黙の型変換を拒否し、プロンプトの組み立てやツール実行の前にサーバー側で入力を検証することを確認してください。 |  1  |  D  |
| 2.4.2 | 最大トークン数またはバイト数の制限を超える入力が、安全なエラーで拒否され、決して無言で切り捨てられないことを確認してください。                                                                                    |  1  | D/V |
| 2.4.3 | ツールやMCPの引数を含め、型チェック（例：数値範囲、列挙値、画像/音声のMIMEタイプ）がサーバー側で確実に適用されていることを検証してください。                                                                         |  2  | D/V |
| 2.4.4 | NLP入力を理解するセマンティックバリデーターが、一定時間で実行され、アルゴリズム的DoSを防ぐために外部ネットワーク呼び出しを回避することを確認してください。                                                                   |  2  |  D  |
| 2.4.5 | 検証の失敗が、伏せ字化されたペイロードの断片と明確なエラーコードとともにログに記録され、セキュリティトリアージを支援するためのトレースメタデータ（ソース、ツールまたはMCPサーバー、エージェントID、セッション）を含むことを確認してください。                          |  3  |  V  |

## C2.5 コンテンツおよびポリシースクリーニング

開発者は、禁止されている内容（違法な指示、ヘイトスピーチ、および/または著作権で保護されたテキストなど）を要求する構文的に有効なプロンプトを検出し、それらが拡散するのを防止できるようにする必要があります。

|   #   | 説明                                                                                                                                                                          | レベル | 役割  |
| :---: | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 2.5.1 | コンテンツ分類器（ゼロショットまたはファインチューニング済み）が、暴力、自傷、ヘイト、性的コンテンツ、および違法なリクエストについて、すべての入力と出力をスコアリングし、しきい値を設定可能であることを確認してください。                                                               |  1  |  D  |
| 2.5.2 | ポリシーに違反する入力は拒否され、下流のLLMやツール/MCP呼び出しに伝播しないことを確認してください。                                                                                                                       |  1  | D/V |
| 2.5.3 | リクエスト時に解決される属性ベースのルール（エージェントの役割属性を含む）を通じて、ユーザー固有のポリシー（年齢、地域の法的制約など）を検閲が遵守していることを検証します。                                                                                      |  2  |  D  |
| 2.5.4 | スクリーニングログに、分類器の信頼度スコアとポリシーカテゴリタグが含まれていることを確認してください。これらには適用されたステージ（プレプロンプトまたはポストレスポンス）およびトレースメタデータ（ソース、ツールまたはMCPサーバー、エージェントID、セッション）が含まれており、SOCの相関分析および将来のレッドチームリプレイに活用されます。 |  3  |  V  |

## C2.6 入力レート制限と悪用防止

開発者は、入力率を制限し、異常な使用パターンを検出することで、AIシステムに対する悪用、リソース枯渇、自動化攻撃を防止すべきです。

|   #   | 説明                                                                                                                                | レベル | 役割  |
| :---: | --------------------------------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 2.6.1 | 全ての入力およびツール/MCPエンドポイントに対して、ユーザー毎、IP毎、APIキー毎、エージェント毎、セッション/タスク毎のレート制限が適用されていることを検証してください。                                          |  1  | D/V |
| 2.6.2 | DoSおよびブルートフォース攻撃を防止するためにバーストおよび持続レート制限が調整されていること、およびエージェントのプランニングループに対してタスクごとの予算（例えば、トークン数、ツール/MCP呼び出し回数、コスト）が適用されていることを確認してください。 |  2  | D/V |
| 2.6.3 | 異常な使用パターン（例：連続的なリクエスト、入力の集中、繰り返し失敗するツール/MCP呼び出し、再帰的なエージェントループ）が自動ブロックまたはエスカレーションを引き起こすことを確認してください。                                |  2  | D/V |
| 2.6.4 | 悪用防止ログが保持され、新たに発生する攻撃パターンを監視するために、トレースメタデータ（送信元、ツールまたはMCPサーバー、エージェントID、セッション）とともにレビューされていることを確認してください。                            |  3  |  V  |

## C2.7 マルチモーダル入力検証

AIシステムは、注入、回避、またはリソースの悪用を防ぐために、非テキスト入力（画像、音声、ファイル）に対して堅牢な検証を含めるべきです。

|   #   | 説明                                                                                                                                                                     | レベル | 役割  |
| :---: | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 2.7.1 | すべての非テキスト入力（画像、音声、ファイル）が処理前にタイプ、サイズ、フォーマットについて検証されていること、また抽出されたテキスト（画像からのテキスト変換や音声からのテキスト変換）や隠された指示（メタデータ、レイヤー、代替テキスト、コメント）がすべて2.1.1に従い信頼されていないものとして扱われていることを確認してください。 |  1  |  D  |
| 2.7.2 | ファイルが取り込み前にマルウェアおよびステガノグラフィックペイロードのスキャンを受けていること、さらにアクティブコンテンツ（スクリプトやマクロなど）が削除されているか、ファイルが隔離されていることを確認してください。                                                           |  2  | D/V |
| 2.7.3 | 画像/音声入力が敵対的摂動や既知の攻撃パターンについて検査されていることを確認し、検出時にはモデル使用前にゲーティング（機能のブロックまたは低下）を発動するようにしてください。                                                                               |  2  | D/V |
| 2.7.4 | マルチモーダル入力検証失敗がログに記録され、調査のためにアラートがトリガーされることを、トレースメタデータ（ソース、ツールまたはMCPサーバー、エージェントID、セッション）とともに確認してください。                                                                   |  3  |  V  |
| 2.7.5 | クロスモーダル攻撃検出が、複数の入力タイプ（例：画像内のステガノグラフィックペイロードとテキスト内のプロンプトインジェクションを組み合わせた攻撃）にまたがる協調攻撃を相関ルールとアラート生成によって識別し、検出が確定した攻撃をブロックするか、HITL（人間の介入）承認を必要とすることを確認してください。               |  2  | D/V |
| 2.7.6 | マルチモーダル検証の失敗が、すべての入力モダリティ、検証結果および脅威スコア、トレースメタデータ（ソース、ツールまたはMCPサーバー、エージェントID、セッション）を含む詳細なログ記録をトリガーすることを確認してください。                                                        |  3  | D/V |

## C2.8 リアルタイム適応型脅威検出

開発者は、新しい攻撃パターンに適応し、コンパイルされたパターンマッチングによるリアルタイム保護を提供する高度なAI向け脅威検出システムを採用すべきです。

|   #   | 説明                                                                                                                       | レベル | 役割  |
| :---: | ------------------------------------------------------------------------------------------------------------------------ | :-: | :-: |
| 2.8.1 | パターンマッチング（例：コンパイル済み正規表現）が、最小限の遅延影響で全ての入力および出力（ツール/MCPの表面を含む）に対して実行されることを検証してください。                                        |  1  | D/V |
| 2.8.3 | 適応型検出モデルが、最近の攻撃活動に基づいて感度を調整し、新しいパターンでリアルタイムに更新されることを確認し、リスク適応型の対応（例えば、ツールの無効化、コンテキストの縮小、またはHITL承認の要求）をトリガーすることを検証してください。 |  2  | D/V |
| 2.8.4 | ユーザーの履歴、ソース、およびセッションの動作（トレースメタデータを含む：ソース、ツールまたはMCPサーバー、エージェントID、セッション）を文脈的に分析することで、検出精度が向上していることを検証してください。               |  3  | D/V |
| 2.8.5 | 検出パフォーマンス指標（検出率、誤検知率、処理遅延時間）が継続的に監視および最適化されていることを確認し、タイム・トゥ・ブロックおよびステージ（プレスプト前/レスポンス後）も含めて行うこと。                          |  3  | D/V |
