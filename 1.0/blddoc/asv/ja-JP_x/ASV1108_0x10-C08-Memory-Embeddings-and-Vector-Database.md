# C8 メモリ、埋め込み、およびベクターデータベースのセキュリティ

## 制御目標

埋め込みおよびベクトルストアは、現代のAIシステムの「ライブメモリ」として機能し、ユーザーから提供されたデータを継続的に受け入れ、Retrieval-Augmented Generation（RAG）を介してモデルのコンテキストに再度取り込む役割を果たします。これが未管理のままだと、個人識別情報（PII）が漏洩したり、同意違反が発生したり、元のテキストを再構築される危険性があります。この管理群の目的は、メモリパイプラインとベクトルデータベースを強化し、アクセス権限を最小限に抑え、埋め込みをプライバシー保護型にし、保存されたベクトルの有効期限を設定または要求に応じて取り消せるようにし、ユーザーごとのメモリが他のユーザーのプロンプトや生成結果に影響を与えないようにすることです。

---

## C8.1 メモリおよびRAGインデックスに対するアクセス制御

すべてのベクターコレクションに対してきめ細かいアクセス制御を強制してください。

|   #   | 説明                                                                                 | レベル | 役割  |
| :---: | ---------------------------------------------------------------------------------- | :-: | :-: |
| 8.1.1 | テナント、コレクション、またはドキュメントタグごとに、行／名前空間レベルのアクセス制御ルールが挿入、削除、およびクエリ操作を制限していることを検証してください。   |  1  | D/V |
| 8.1.2 | APIキーまたはJWTがスコープ付きクレーム（例：コレクションID、アクション動詞）を持ち、少なくとも四半期ごとにローテーションされていることを確認してください。  |  1  | D/V |
| 8.1.3 | 特権昇格の試み（例：クロスネームスペース類似性クエリ）が検出され、5分以内にSIEMにログが記録されることを確認してください。                    |  2  | D/V |
| 8.1.4 | ベクターデータベースがログに記録する項目として、サブジェクト識別子、操作内容、ベクターID／名前空間、類似度閾値、および結果数を監査していることを確認してください。 |  2  | D/V |
| 8.1.5 | エンジンがアップグレードされるか、インデックスシャーディングルールが変更されるたびに、アクセス決定がバイパスの欠陥についてテストされていることを検証してください。  |  3  |  V  |

## C8.2 埋め込みのサニタイズと検証

PIIを事前にスクリーニングし、ベクトル化の前に検閲または仮名化を行い、必要に応じて埋め込み後処理で残存する信号を除去します。

|   #   | 説明                                                                                     | レベル | 役割  |
| :---: | -------------------------------------------------------------------------------------- | :-: | :-: |
| 8.2.1 | PIIおよび規制対象データが自動分類器によって検出され、埋め込み前にマスク処理、トークン化、または除去されることを確認してください。                     |  1  | D/V |
| 8.2.2 | 埋め込みパイプラインが、インデックスを汚染する可能性のある実行可能コードや非UTF-8アーティファクトを含む入力を拒否または隔離することを検証してください。         |  1  |  D  |
| 8.2.3 | 既知のPIIトークンからの距離が設定可能な閾値を下回る文埋め込みに対して、局所的またはメトリックの差分プライバシーサニタイゼーションが適用されていることを検証してください。 |  2  | D/V |
| 8.2.4 | サニタイズの有効性（例：PII削除の再現率、意味のずれ）が、ベンチマークコーパスに対して少なくとも半年に一度検証されていることを確認してください。              |  2  |  V  |
| 8.2.5 | サニタイズ設定がバージョン管理されており、変更がピアレビューを経ていることを確認してください。                                        |  3  | D/V |

## C8.3 メモリの有効期限切れ、取り消し、および削除

GDPRの「忘れられる権利」や類似の法律は迅速な消去を要求しているため、ベクターストアはTTL、ハードデリート、およびトゥームストーニングをサポートし、取り消されたベクターが回復または再インデックス化されないようにする必要があります。

|   #   | 説明                                                                            | レベル | 役割  |
| :---: | ----------------------------------------------------------------------------- | :-: | :-: |
| 8.3.1 | すべてのベクターおよびメタデータレコードがTTLまたは自動クリーンアップジョブによって尊重される明示的な保持ラベルを持っていることを確認してください。   |  1  | D/V |
| 8.3.2 | ユーザーによる削除要求が行われた場合、30日以内にベクトル、メタデータ、キャッシュコピー、および派生インデックスが完全に削除されることを確認してください。 |  1  | D/V |
| 8.3.3 | 論理削除の後に、ハードウェアが対応している場合はストレージブロックの暗号的削除、またはキーボルトキーの破棄が行われていることを確認してください。      |  2  |  D  |
| 8.3.4 | 有効期限切れのベクトルが、有効期限切れから500ms未満で最近傍検索結果から除外されていることを確認してください。                     |  3  | D/V |

## C8.4 埋め込みの逆転と漏洩の防止

最近の防御策—ノイズ重畳、射影ネットワーク、プライバシーニューロン摂動、およびアプリケーション層の暗号化—により、トークンレベルの逆転率を5%以下に抑えることが可能です。

|   #   | 説明                                                                                             | レベル | 役割  |
| :---: | ---------------------------------------------------------------------------------------------- | :-: | :-: |
| 8.4.1 | 反転攻撃、メンバーシップ攻撃、属性推測攻撃をカバーする正式な脅威モデルが存在し、かつ毎年レビューされていることを検証する。                                  |  1  |  V  |
| 8.4.2 | アプリケーション層の暗号化または検索可能な暗号化が、インフラ管理者やクラウドスタッフによるベクトルの直接読み取りから保護していることを検証してください。                   |  2  | D/V |
| 8.4.3 | 防御パラメータ（差分プライバシーのためのε、ノイズのσ、射影ランクのk）が、プライバシー≥99％のトークン保護とユーティリティ≤3％の精度損失のバランスを保っていることを確認してください。 |  3  |  V  |
| 8.4.4 | モデル更新のリリースゲートに反転耐性メトリクスが組み込まれており、回帰予算が定義されていることを確認してください。                                      |  3  | D/V |

## C8.5 ユーザー固有メモリのスコープ強制

クロステナントの情報漏えいは依然として主要なRAGリスクです。不適切にフィルタリングされた類似性クエリが別の顧客のプライベートドキュメントを表示する可能性があります。

|   #   | 説明                                                                               | レベル | 役割  |
| :---: | -------------------------------------------------------------------------------- | :-: | :-: |
| 8.5.1 | すべての検索クエリがLLMプロンプトに渡される前に、テナント/ユーザーIDによってポストフィルタリングされることを確認してください。               |  1  | D/V |
| 8.5.2 | コレクション名または名前空間付きIDが、ベクトルがスコープ間で衝突しないようにユーザーまたはテナントごとにソルトされていることを確認してください。        |  1  |  D  |
| 8.5.3 | 設定可能な距離しきい値を超え、かつ呼び出し元の範囲外にある類似性の結果が破棄され、セキュリティアラートが発生することを確認してください。             |  2  | D/V |
| 8.5.4 | マルチテナントのストレステストが、スコープ外の文書を取得しようとする敵対的なクエリをシミュレートし、情報漏洩が全くないことを実証していることを確認してください。 |  2  |  V  |
| 8.5.5 | 暗号鍵がテナントごとに分離されていることを確認し、物理的なストレージが共有されている場合でも暗号的な分離が確保されていることを保証してください。         |  3  | D/V |

## C8.6 高度なメモリシステムのセキュリティ

エピソード記憶、意味記憶、ワーキングメモリを含む高度なメモリアーキテクチャに対する、特定の分離および検証要件を伴うセキュリティコントロール。

|   #   | 説明                                                                                                                    | レベル | 役割  |
| :---: | --------------------------------------------------------------------------------------------------------------------- | :-: | :-: |
| 8.6.1 | 異なるメモリタイプ（エピソード記憶、意味記憶、作業記憶）が、ロールベースアクセス制御、個別の暗号化キー、および各メモリタイプごとに文書化されたアクセスパターンを持つ分離されたセキュリティコンテキストを持っていることを検証してください。 |  1  | D/V |
| 8.6.2 | メモリ統合プロセスに、コンテンツのサニタイズ、ソースの検証、および保存前の整合性チェックを通じて悪意のあるメモリの注入を防ぐためのセキュリティ検証が含まれていることを確認する。                              |  2  | D/V |
| 8.6.3 | メモリ取得クエリが検証およびサニタイズされ、不正な情報抽出を防ぐために、クエリパターンの解析、アクセス制御の強制、結果のフィルタリングが行われていることを確認してください。                                |  2  | D/V |
| 8.6.4 | キー削除、多重上書き、または検証証明書付きのハードウェアベースの安全な削除を使用して、暗号消去の保証とともに機密情報が確実に削除されることを検証します。                                          |  3  | D/V |
| 8.6.5 | メモリシステムの整合性が、チェックサム、監査ログ、および通常の操作外でメモリ内容が変更された場合の自動アラートを通じて、不正な変更や破損について継続的に監視されていることを確認してください。                       |  3  | D/V |
