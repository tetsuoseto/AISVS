# การตรวจสอบความถูกต้องของข้อมูลที่ผู้ใช้ C2 ป้อน

## วัตถุประสงค์ในการควบคุม

การตรวจสอบความถูกต้องของข้อมูลที่ผู้ใช้ป้อนอย่างเข้มงวดเป็นแนวป้องกันแรกที่สำคัญต่อการโจมตีที่สร้างความเสียหายอย่างรุนแรงต่อระบบปัญญาประดิษฐ์ การโจมตีด้วยการแทรกคำสั่ง (prompt injection) สามารถเปลี่ยนแปลงคำสั่งของระบบ เปิดเผยข้อมูลที่ละเอียดอ่อน หรือชี้นำโมเดลไปสู่พฤติกรรมที่ไม่ได้รับอนุญาตได้ เว้นแต่จะมีตัวกรองและการตรวจสอบความถูกต้องเฉพาะทาง การวิจัยแสดงให้เห็นว่าการเจลเบรกที่ใช้ประโยชน์จากหน้าต่างบริบทจะยังคงมีประสิทธิภาพอยู่ต่อไป

---

## C2.1 การป้องกันการฉีดพรอมต์

การโจมตีด้วยการฝังคำสั่ง (Prompt injection) เป็นหนึ่งในความเสี่ยงสูงสุดสำหรับระบบ AI การป้องกันวิธีนี้ใช้การผสมผสานของตัวกรองรูปแบบ ตัวจำแนกข้อมูล และการบังคับใช้ลำดับขั้นของคำสั่ง

|   #   | คำอธิบาย                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           | ระดับ | บทบาท |
| :---: | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---: | :---: |
| 2.1.1 | ตรวจสอบให้แน่ใจว่าข้อมูลนำเข้าภายนอกหรือข้อมูลที่ได้มาซึ่งอาจมีผลต่อพฤติกรรมต่างๆ รวมถึงคำสั่งของผู้ใช้ ผลลัพธ์ RAG ผลลัพธ์จากปลั๊กอินหรือ MCP ข้อความระหว่างตัวแทนไปยังตัวแทน การตอบกลับ API หรือ webhook ไฟล์การตั้งค่าหรือไฟล์นโยบาย การอ่านหรือการเขียนหน่วยความจำ ถูกปฏิบัติอย่างไม่ไว้วางใจ ถูกทำให้เป็นกลางโดยการใส่เครื่องหมายคำพูดหรือแท็กและการลบเนื้อหาที่เป็นเชิงรุก และถูกคัดกรองด้วยชุดกฎหรือบริการตรวจจับการฉีดคำสั่งที่ได้รับการบำรุงรักษาก่อนที่จะนำไปรวมในคำสั่งหรือดำเนินการใดๆ |   1   |  D/V  |
| 2.1.2 | ตรวจสอบให้แน่ใจว่าระบบบังคับใช้ลำดับชั้นคำสั่งซึ่งในนั้นข้อความของระบบและผู้พัฒนาจะมีสิทธิ์เหนือกว่าคำสั่งของผู้ใช้และข้อมูลนำเข้าที่ไม่น่าเชื่อถืออื่น ๆ แม้หลังจากประมวลผลคำสั่งของผู้ใช้แล้วก็ตาม                                                                                                                                                                                                                                                                                               |   1   |  D/V  |
| 2.1.4 | ตรวจสอบให้แน่ใจว่า prompt ที่มาจากเนื้อหาของบุคคลที่สาม (หน้าเว็บ, PDF, อีเมล) ถูกทำความสะอาดอย่างแยกต่างหาก (เช่น การลบคำสั่งที่คล้ายกับคำแนะนำและการทำให้เนื้อหา HTML, Markdown, และสคริปต์เป็นกลาง) ก่อนที่จะนำมาต่อรวมกับ prompt หลัก                                                                                                                                                                                                                                                          |   2   |   D   |

## C2.2 ความต้านทานต่อการโจมตีด้วยตัวอย่างคู่แข่ง

โมเดลการประมวลผลภาษาธรรมชาติ (NLP) ยังมีความอ่อนแอต่อการรบกวนเล็กน้อยในระดับตัวอักษรหรือคำที่มนุษย์มักจะมองข้าม แต่โมเดลมักจะจัดประเภทผิดพลาด

|   #   | คำอธิบาย                                                                                                                                                                                                                                                                                                                                                         | ระดับ | บทบาท |
| :---: | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---: | :---: |
| 2.2.1 | ตรวจสอบให้แน่ใจว่าขั้นตอนการปรับมาตรฐานอินพุตพื้นฐาน (Unicode NFC, การแมปโฮโมกลิฟ, การตัดช่องว่าง, การลบอักขระควบคุมและอักขระ Unicode ที่มองไม่เห็น) ถูกดำเนินการก่อนการตัดคำหรือการฝังข้อมูล และก่อนการแยกวิเคราะห์เป็นอาร์กิวเมนต์ของเครื่องมือหรือ MCP                                                                                                        |   1   |   D   |
| 2.2.2 | ตรวจสอบให้แน่ใจว่าการตรวจจับความผิดปกติทางสถิติสามารถตรวจจับข้อมูลนำเข้าที่มีระยะห่างการแก้ไขสูงผิดปกติต่อมาตรฐานภาษา หรือระยะห่างการฝังตัวที่ผิดปกติ และข้อมูลนำเข้าที่ถูกตรวจจับไว้จะถูกคัดกรองก่อนการเชื่อมต่อเข้ากับคำสั่งนำหรือการดำเนินการต่อไป                                                                                                            |   2   |  D/V  |
| 2.2.3 | ตรวจสอบให้แน่ใจว่าโครงงานอนุมานรองรับแบบจำลองที่ได้รับการฝึกฝนเชิงป้องกันด้วยการโจมตีหรือชั้นป้องกัน (เช่น การสุ่ม การกลั่นกรองเชิงป้องกัน การตรวจสอบความสอดคล้อง) สำหรับจุดสิ้นสุดที่มีความเสี่ยงสูง                                                                                                                                                            |   2   |   D   |
| 2.2.4 | ตรวจสอบให้แน่ใจว่าสิ่งที่คาดว่าจะเป็นอินพุตก่อกวนถูกกักบริเวณ และบันทึกไว้พร้อมด้วย payload เต็มรูปแบบและเมตาดาต้าร่องรอย (แหล่งที่มา เครื่องมือหรือเซิร์ฟเวอร์ MCP รหัสตัวแทน เซสชัน)                                                                                                                                                                           |   2   |   V   |
| 2.2.5 | ตรวจสอบให้แน่ใจว่าการลักลอบแฝงในการเข้ารหัสและการแทนค่าทั้งในข้อมูลนำเข้าและข้อมูลส่งออก (เช่น อักขระ Unicode/ควบคุมที่มองไม่เห็น, การสลับฮโมโกลิฟ, หรือข้อความทิศทางผสม) ถูกตรวจจับและลดความเสี่ยงแล้ว มาตรการลดความเสี่ยงที่ได้รับอนุมัติรวมถึงการทำให้เป็นรูปแบบมาตรฐาน, การตรวจสอบตามแบบแผนอย่างเข้มงวด, การปฏิเสธตามนโยบาย, หรือการทำเครื่องหมายอย่างชัดเจน |   2   |  D/V  |

## C2.3 ชุดตัวอักษรพรอมต์

การจำกัดชุดตัวอักษรของข้อมูลที่ผู้ใช้ป้อนให้มีเฉพาะตัวอักษรที่จำเป็นสำหรับความต้องการทางธุรกิจเท่านั้น สามารถช่วยป้องกันการโจมตีประเภทต่างๆ ได้

|   #   | คำอธิบาย                                                                                                                                                        | ระดับ | บทบาท |
| :---: | --------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---: | :---: |
| 2.3.1 | ตรวจสอบให้แน่ใจว่าระบบมีการจำกัดชุดอักขระสำหรับข้อมูลนำเข้าของผู้ใช้ โดยอนุญาตเฉพาะอักขระที่จำเป็นอย่างชัดเจนสำหรับวัตถุประสงค์ทางธุรกิจเท่านั้น                |   1   |   D   |
| 2.3.2 | ตรวจสอบให้แน่ใจว่าใช้วิธีการแบบรายการอนุญาต (allow-list) เพื่อกำหนดชุดตัวอักษรที่ได้รับอนุญาต                                                                   |   1   |   D   |
| 2.3.3 | ตรวจสอบให้แน่ใจว่าข้อมูลนำเข้าที่มีอักขระนอกชุดที่อนุญาตถูกปฏิเสธและบันทึกพร้อมข้อมูลเมตาแบบติดตาม (แหล่งที่มา เครื่องมือหรือเซิร์ฟเวอร์ MCP รหัสตัวแทน เซสชัน) |   1   |  D/V  |

## C2.4 การตรวจสอบสคีมา ประเภท และความยาว

การโจมตีด้วย AI ที่มีอินพุตที่บิดเบี้ยวหรือมีขนาดใหญ่เกินไปสามารถทำให้เกิดข้อผิดพลาดในการแยกวิเคราะห์ข้อมูล การรั่วไหลของคำสั่งระหว่างฟิลด์ และการใช้ทรัพยากรจนหมด การบังคับใช้สคีมาอย่างเข้มงวดก็เป็นข้อกำหนดเบื้องต้นเมื่อทำการเรียกใช้เครื่องมือแบบมีการกำหนดผลลัพธ์ที่แน่นอน

|   #   | คำอธิบาย                                                                                                                                                                                                                                                                                                     | ระดับ | บทบาท |
| :---: | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | :---: | :---: |
| 2.4.1 | ตรวจสอบให้แน่ใจว่า API, เครื่องมือ หรือจุดเชื่อมต่อ MCP ทุกตัวกำหนดสคีมาอินพุตอย่างชัดเจน (JSON Schema, Protobuf หรือเทียบเท่ามัลติช่องทาง) ปฏิเสธฟิลด์ที่เกินหรือไม่รู้จักและการบังคับแปลงชนิดข้อมูลโดยนัย และตรวจสอบความถูกต้องของอินพุตที่ฝั่งเซิร์ฟเวอร์ก่อนการประกอบพรอมต์หรือการดำเนินการของเครื่องมือ |   1   |   D   |
| 2.4.2 | ตรวจสอบให้แน่ใจว่าอินพุตที่เกินขีดจำกัดโทเค็นหรือไบต์สูงสุดจะถูกปฏิเสธด้วยข้อผิดพลาดที่ปลอดภัยและไม่ถูกตัดทอนแบบเงียบๆ                                                                                                                                                                                       |   1   |  D/V  |
| 2.4.3 | ตรวจสอบให้แน่ใจว่าการตรวจสอบชนิดข้อมูล (เช่น ช่วงตัวเลข ค่าของ enum ประเภท MIME สำหรับภาพ/เสียง) ถูกบังคับใช้ที่ฝั่งเซิร์ฟเวอร์ รวมถึงสำหรับอาร์กิวเมนต์ของเครื่องมือหรือ MCP ด้วย                                                                                                                           |   2   |  D/V  |
| 2.4.4 | ตรวจสอบให้แน่ใจว่า semantic validators ที่เข้าใจข้อมูลภาษา NLP ทำงานในเวลาคงที่และหลีกเลี่ยงการเรียกเครือข่ายภายนอกเพื่อป้องกันการโจมตีแบบ DoS ทางอัลกอริทึม                                                                                                                                                 |   2   |   D   |
| 2.4.5 | ตรวจสอบให้แน่ใจว่าความล้มเหลวในการตรวจสอบถูกบันทึกโดยมีการตัดข้อมูลบางส่วนของ payload ออกและรหัสข้อผิดพลาดที่ชัดเจน รวมทั้งมีข้อมูลเมตาแทรซ (แหล่งที่มา เครื่องมือหรือเซิร์ฟเวอร์ MCP รหัสตัวแทน เซสชัน) เพื่อช่วยในการจัดการความปลอดภัย                                                                     |   3   |   V   |

## C2.5 การกรองเนื้อหาและนโยบาย

นักพัฒนาควรสามารถตรวจจับคำสั่งที่ถูกต้องตามไวยากรณ์ซึ่งร้องขอเนื้อหาที่ไม่ได้รับอนุญาต (เช่น คำแนะนำที่ผิดกฎหมาย คำพูดแสดงความเกลียดชัง และ/หรือข้อความที่มีลิขสิทธิ์) แล้วป้องกันไม่ให้คำสั่งเหล่านั้นแพร่กระจายออกไปได้

|   #   | คำอธิบาย                                                                                                                                                                                                                                                                                      | ระดับ | บทบาท |
| :---: | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---: | :---: |
| 2.5.1 | ตรวจสอบให้แน่ใจว่าตัวจัดประเภทเนื้อหา (แบบ zero shot หรือแบบปรับแต่งเฉพาะ) ให้คะแนนทุกข้อมูลเข้าและออกสำหรับความรุนแรง การทำร้ายตนเอง ความเกลียดชัง เนื้อหาเกี่ยวกับเพศ และคำขอที่ผิดกฎหมาย โดยมีเกณฑ์ที่ปรับแต่งได้                                                                          |   1   |   D   |
| 2.5.2 | ตรวจสอบให้แน่ใจว่าข้อมูลนำเข้าที่ละเมิดนโยบายจะถูกปฏิเสธเพื่อไม่ให้ถูกส่งต่อไปยังการเรียกใช้ LLM หรือเครื่องมือ/MCP ที่อยู่ในลำดับถัดไป                                                                                                                                                       |   1   |  D/V  |
| 2.5.3 | ตรวจสอบให้แน่ใจว่าการคัดกรองเป็นไปตามนโยบายเฉพาะผู้ใช้ (อายุ ข้อจำกัดทางกฎหมายในภูมิภาค) ผ่านกฎที่อิงกับคุณลักษณะซึ่งแก้ไขได้ในเวลาที่มีการร้องขอ รวมถึงคุณลักษณะบทบาทของตัวแทนด้วย                                                                                                           |   2   |   D   |
| 2.5.4 | ตรวจสอบให้แน่ใจว่าบันทึกการคัดกรองมีคะแนนความมั่นใจของตัวจำแนกและแท็กหมวดหมู่นโยบายพร้อมกับระยะเวลาที่นำไปใช้ (ก่อน-พรอมท์ หรือ หลัง-การตอบกลับ) รวมถึงข้อมูลติดตามการทำงาน (แหล่งที่มา เครื่องมือหรือเซิร์ฟเวอร์ MCP รหัสตัวแทน เซสชัน) สำหรับการเชื่อมโยง SOC และการเล่นซ้ำของทีมแดงในอนาคต |   3   |   V   |

## C2.6 การจำกัดอัตราการป้อนข้อมูลและการป้องกันการละเมิด

นักพัฒนาควรป้องกันการใช้ในทางที่ผิด การใช้ทรัพยากรจนหมด และการโจมตีอัตโนมัติต่อระบบ AI โดยการจำกัดอัตราการป้อนข้อมูลและตรวจจับรูปแบบการใช้งานที่ผิดปกติ

| :---: | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---: | :---: |
| 2.6.1 | ตรวจสอบให้แน่ใจว่าการจำกัดอัตราการใช้งานต่อผู้ใช้แต่ละราย ต่อ IP แต่ละรายการ ต่อคีย์ API แต่ละรายการ ต่อเอเจนต์แต่ละตัว และต่อเซสชัน/งานแต่ละงาน ได้รับการบังคับใช้สำหรับจุดเชื่อมต่อของข้อมูลนำเข้าและเครื่องมือ/MCP ทั้งหมดแล้ว                                                       |   1   |  D/V  |

| 2.6.3 | ตรวจสอบให้แน่ใจว่ารูปแบบการใช้งานที่ผิดปกติ (เช่น คำขอที่ส่งอย่างรวดเร็ว, การป้อนข้อมูลจำนวนมากอย่างต่อเนื่อง, การเรียกใช้เครื่องมือ/MCP ที่ล้มเหลวซ้ำๆ หรือการวนลูปของตัวแทนแบบเรียกซ้ำ) จะทำให้เกิดการบล็อกอัตโนมัติหรือการยกระดับการจัดการ                                           |   2   |  D/V  |

## C2.7 การตรวจสอบความถูกต้องของอินพุตแบบมัลติ-โหมด

ระบบ AI ควรรวมการตรวจสอบความถูกต้องที่แข็งแกร่งสำหรับข้อมูลนำเข้าแบบไม่ใช่ข้อความ (ภาพ, เสียง, ไฟล์) เพื่อป้องกันการฉีดข้อมูล, การหลบเลี่ยง, หรือการใช้ทรัพยากรเกินจำเป็น.

|   #   | คำอธิบาย                                                                                                                                                                                                                                                                                                                                                         | ระดับ | บทบาท |

| 2.7.1 | ตรวจสอบให้แน่ใจว่าข้อมูลนำเข้าที่ไม่ใช่ข้อความทั้งหมด (ภาพ, เสียง, ไฟล์) ได้รับการตรวจสอบประเภท, ขนาด, และรูปแบบอย่างถูกต้องก่อนการประมวลผล และข้อความใดๆ ที่ถูกดึงออกมา (จากภาพเป็นข้อความหรือจากเสียงเป็นข้อความ) รวมถึงคำสั่งที่ซ่อนอยู่หรือฝังอยู่ (เมทาดาต้า, ชั้นข้อมูล, ข้อความแทนภาพ, ความคิดเห็น) จะถูกจัดการว่าเป็นข้อมูลที่ไม่เชื่อถือได้ตามข้อ 2.1.1 |   1   |   D   |
| 2.7.2 | ตรวจสอบให้แน่ใจว่าไฟล์ถูกสแกนหาไวรัสและ payload แบบสเตโกโนกราฟิกก่อนนำเข้า และว่ามีการลบเนื้อหาแอคทีฟ (เช่น สคริปต์หรือมาโคร) หรือกักกันไฟล์ไว้แล้ว                                                                                                                                                                                                              |   2   |  D/V  |
| 2.7.3 | ตรวจสอบให้แน่ใจว่าข้อมูลนำเข้าเป็นภาพ/เสียงได้รับการตรวจสอบหาการรบกวนจากฝ่ายตรงข้ามหรือรูปแบบการโจมตีที่รู้จัก และเมื่อมีการตรวจพบจะกระตุ้นการควบคุม (บล็อกหรือจำกัดความสามารถ) ก่อนการใช้งานโมเดล                                                                                                                                                               |   2   |  D/V  |
| 2.7.4 | ตรวจสอบให้แน่ใจว่าความล้มเหลวในการตรวจสอบข้อมูลนำเข้ารูปแบบมัลติ-โหมดถูกบันทึกและทำให้เกิดการแจ้งเตือนเพื่อการสอบสวน พร้อมเมตาดาต้าการติดตาม (แหล่งที่มา เครื่องมือหรือเซิร์ฟเวอร์ MCP รหัสตัวแทน เซสชัน)                                                                                                                                                        |   3   |   V   |
| 2.7.5 | ตรวจสอบให้แน่ใจว่าการตรวจจับการโจมตีข้ามรูปแบบสามารถระบุการโจมตีที่ประสานงานกันซึ่งครอบคลุมหลายประเภทข้อมูลนำเข้า (เช่น พาหะซ่อนเร้นในรูปภาพร่วมกับการฉีดคำสั่งในข้อความ) โดยใช้กฎความสัมพันธ์และการสร้างการแจ้งเตือน และการตรวจจับที่ได้รับการยืนยันจะถูกบล็อกหรือต้องได้รับการอนุมัติจากผู้ควบคุมมนุษย์ (HITL)                                                 |   2   |  D/V  |
| 2.7.6 | ตรวจสอบให้แน่ใจว่าความล้มเหลวในการตรวจสอบแบบมัลติ-โหมดจะทำให้เกิดการบันทึกรายละเอียดอย่างละเอียด รวมถึงโหมดอินพุตทั้งหมด ผลการตรวจสอบ และคะแนนภัยคุกคาม รวมทั้งเมตาดาต้าแทรซ (แหล่งที่มา เครื่องมือหรือเซิร์ฟเวอร์ MCP รหัสตัวแทน เซสชัน)                                                                                                                        |   3   |  D/V  |

## C2.8 การตรวจจับภัยคุกคามแบบปรับตัวเรียลไทม์

นักพัฒนาควรใช้ระบบตรวจจับภัยคุกคามขั้นสูงสำหรับ AI ที่ปรับตัวกับรูปแบบการโจมตีใหม่ๆ และให้การป้องกันแบบเรียลไทม์โดยใช้การจับคู่รูปแบบที่รวบรวมแล้ว

| :---: | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | :---: | :---: |
| 2.8.1 | ตรวจสอบให้แน่ใจว่าการจับคู่รูปแบบ (เช่น regex ที่ถูกคอมไพล์) ทำงานบนข้อมูลนำเข้าและข้อมูลส่งออกทั้งหมด (รวมถึงเครื่องมือ/พื้นที่ MCP) โดยมีผลกระทบต่อความหน่วงเวลาน้อยที่สุด                                                                                 |   1   |  D/V  |
| 2.8.3 | ตรวจสอบให้แน่ใจว่าโมเดลการตรวจจับแบบปรับตัวจะปรับความไวตามกิจกรรมการโจมตีล่าสุดและได้รับการอัปเดตด้วยรูปแบบใหม่แบบเรียลไทม์ รวมทั้งกระตุ้นการตอบสนองที่ปรับตามความเสี่ยง (เช่น ปิดการใช้งานเครื่องมือ ลดขอบเขตบริบท หรือเรียกการอนุมัติจากมนุษย์ในการควบคุม) |   2   |  D/V  |
