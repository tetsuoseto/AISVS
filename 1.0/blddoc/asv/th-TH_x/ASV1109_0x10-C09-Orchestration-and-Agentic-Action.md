# 9 การจัดการอัตโนมัติอย่างอิสระและความปลอดภัยของการดำเนินการแบบตัวแทน

## วัตถุประสงค์ในการควบคุม

ตรวจสอบให้แน่ใจว่าระบบ AI อัตโนมัติหรือหลายตัวแทนสามารถดำเนินการได้เฉพาะการกระทำที่ตั้งใจไว้อย่างชัดเจน มีการยืนยันตัวตน สามารถตรวจสอบย้อนหลังได้ และอยู่ภายในขอบเขตของค่าใช้จ่ายและเกณฑ์ความเสี่ยงที่กำหนดไว้ สิ่งนี้ช่วยป้องกันภัยคุกคาม เช่น การบุกรุกระบบอัตโนมัติ การใช้เครื่องมือผิดวัตถุประสงค์ การตรวจจับวงจรตัวแทน การแฮ็กการสื่อสาร การปลอมแปลงตัวตน การจัดการฝูง และการบิดเบือนเจตนา

## 9.1 งบประมาณการวางแผนงานของเอเย่นต์และการเรียกซ้ำ

| 9.1.2 | ตรวจสอบให้แน่ใจว่าการกระทำที่มีสิทธิพิเศษหรือไม่สามารถย้อนกลับได้ (เช่น การส่งโค้ด, การโอนเงิน) จำเป็นต้องได้รับการอนุมัติจากมนุษย์อย่างชัดเจนผ่านช่องทางที่สามารถตรวจสอบได้ก่อนการดำเนินการ |   1   |  D/V  |

| 9.1.4 | ตรวจสอบให้แน่ใจว่าเหตุการณ์เบรกเกอร์วงจรถูกบันทึกด้วยรหัสตัวแทน, เงื่อนไขที่ทำให้เกิดเหตุการณ์, และสถานะแผนที่ถูกบันทึกสำหรับการตรวจสอบทางนิติเวช                                            |   2   |  D/V  |
| 9.1.5 | ตรวจสอบให้แน่ใจว่าแบบทดสอบด้านความปลอดภัยครอบคลุมกรณีการใช้งบประมาณหมดและแผนการทำงานเกินขอบเขต โดยยืนยันการหยุดทำงานอย่างปลอดภัยโดยไม่สูญเสียข้อมูล                                          |   3   |   V   |
| 9.1.6 | ตรวจสอบให้แน่ใจว่านโยบายงบประมาณถูกแสดงเป็นนโยบายในรูปแบบโค้ดและบังคับใช้ใน CI/CD เพื่อป้องกันการเบี่ยงเบนของการกำหนดค่า                                                                     |   3   |   D   |

| :---: | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---: | :---: |

| 9.2.4 | ตรวจสอบให้แน่ใจว่าการส่งข้อมูลโทรเมตริกของ sandbox ไปยัง SIEM; ความผิดปกติ (เช่น การพยายามเชื่อมต่อขาออก) จะทำให้เกิดการแจ้งเตือน                                                              |   2   |   V   |
| 9.2.5 | ตรวจสอบให้แน่ใจว่า ปลั๊กอินที่มีความเสี่ยงสูงผ่านการทบทวนด้านความปลอดภัยและการทดสอบเจาะระบบก่อนนำไปใช้งานจริงในสภาพแวดล้อมการผลิต                                                              |   3   |   V   |
| 9.2.6 | ตรวจสอบให้แน่ใจว่าการพยายามหลบหนีจาก sandbox ถูกบล็อกโดยอัตโนมัติ และปลั๊กอินที่ทำผิดถูกกักกันรอการตรวจสอบ                                                                                     |   3   |  D/V  |

| 9.3.2 | ตรวจสอบให้นายหน้ารักษา ID กราฟการเรียกใช้งานเฉพาะตัวเพื่อระบุการเรียกใช้งานตนเองหรือรูปแบบวงจร                                       |   2   |   D   |

| 9.3.4 | ตรวจสอบให้แน่ใจว่าการวิเคราะห์เชิงรูปแบบหรือการตรวจสอบแบบโมเดลแสดงให้เห็นว่าไม่มีการเรียกซ้ำแบบไม่จำกัดในโพรโตคอลของตัวแทน           |   3   |   V   |
| 9.3.5 | ตรวจสอบให้แน่ใจว่าเหตุการณ์หยุดการทำงานของลูปสามารถสร้างการแจ้งเตือนและส่งข้อมูลเมตริกสำหรับการปรับปรุงอย่างต่อเนื่องได้             |   3   |   D   |

| :---: | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---: | :---: |

| 9.4.3 | ตรวจสอบให้แน่ใจว่าการตรวจสอบความสมบูรณ์ (MACs หรือ ลายเซ็นดิจิทัล) ครอบคลุมทั้งข้อความที่ส่งข้อมูลรวมถึงพารามิเตอร์ของเครื่องมือด้วย                                |   2   |  D/V  |
| 9.4.4 | ตรวจสอบให้แน่ใจว่าการป้องกันการเล่นซ้ำ (nonce หรือหน้าต่างเวลาของ timestamp) ถูกบังคับใช้ที่ชั้นโปรโตคอล                                                            |   2   |   D   |

| 9.6.3 | ตรวจสอบให้แน่ใจว่าส่วนตรวจสอบในช่วงเวลารันไทม์สามารถตรวจจับรูปแบบไม่ปลอดภัยที่เกิดขึ้นใหม่ (เช่น การสั่น การเกิด deadlock) และเริ่มดำเนินการแก้ไขได้อย่างถูกต้อง |   3   |   D   |

## 9.12 กรอบการทำงานเพื่อความปลอดภัยของการรวมเครื่องมือ

การควบคุมความปลอดภัยสำหรับการโหลดเครื่องมือแบบไดนามิก การดำเนินการ และการตรวจสอบผลลัพธ์พร้อมกับกระบวนการประเมินความเสี่ยงและการอนุมัติที่กำหนดไว้

| :----: | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---: | :---: |
| 9.12.1 | ตรวจสอบให้แน่ใจว่าคำอธิบายเครื่องมือมีเมตาดาต้าด้านความปลอดภัยระบุสิทธิ์ที่จำเป็น (อ่าน/เขียน/ประมวลผล), ระดับความเสี่ยง (ต่ำ/กลาง/สูง), ขีดจำกัดทรัพยากร (CPU, หน่วยความจำ, เครือข่าย) และข้อกำหนดการตรวจสอบที่จัดทำเป็นเอกสารในแผนผังเครื่องมือ               |   1   |  D/V  |
| 9.12.2 | ตรวจสอบให้แน่ใจว่าผลลัพธ์ของการดำเนินการเครื่องมือนั้นได้รับการยืนยันความถูกต้องตามสคีมา (JSON Schema, XML Schema) และนโยบายความปลอดภัย (การกรองผลลัพธ์, การจำแนกประเภทข้อมูล) ก่อนการรวมเข้าด้วยกันพร้อมกับการจำกัดเวลาและขั้นตอนการจัดการข้อผิดพลาด           |   1   |  D/V  |
| 9.12.3 | ตรวจสอบให้แน่ใจว่าบันทึกการโต้ตอบของเครื่องมือรวมบริบทความปลอดภัยอย่างละเอียด เช่น การใช้สิทธิ์ รูปแบบการเข้าถึงข้อมูล เวลาในการดำเนินการ การใช้ทรัพยากร และรหัสผลลัพธ์ พร้อมกับการบันทึกเป็นโครงสร้างสำหรับการรวมระบบ SIEM                                     |   2   |  D/V  |
| 9.12.4 | ตรวจสอบให้แน่ใจว่า กลไกการโหลดเครื่องมือแบบไดนามิก ทำการตรวจสอบลายเซ็นดิจิทัลโดยใช้โครงสร้างพื้นฐาน PKI และนำโปรโตคอลการโหลดที่ปลอดภัยมาใช้ พร้อมทั้งแยกการทำงานใน sandbox และตรวจสอบสิทธิ์ก่อนการใช้งาน                                                        |   2   |  D/V  |
| 9.12.5 | ตรวจสอบให้แน่ใจว่าการประเมินความปลอดภัยของเครื่องมือถูกเปิดใช้งานโดยอัตโนมัติสำหรับเวอร์ชันใหม่โดยมีเกตการอนุมัติที่จำเป็นซึ่งรวมถึงการวิเคราะห์แบบสแตติก การทดสอบแบบไดนามิก และการตรวจสอบโดยทีมรักษาความปลอดภัยโดยมีเกณฑ์การอนุมัติที่บันทึกไว้และข้อกำหนด SLA |   3   |  D/V  |
