# ความปลอดภัยของหน่วยความจำ C8, การฝังข้อมูล และฐานข้อมูลเวกเตอร์

## วัตถุประสงค์ในการควบคุม

การฝังข้อมูลและฐานข้อมูลเวกเตอร์ทำหน้าที่เป็น "ความทรงจำสด" ของระบบ AI ร่วมสมัย โดยรับข้อมูลที่ผู้ใช้ส่งมาอย่างต่อเนื่องและนำกลับมาใช้ในบริบทของโมเดลผ่านการสร้างเสริมการดึงข้อมูล (Retrieval-Augmented Generation หรือ RAG) หากปล่อยให้ไม่มีการควบคุม ความทรงจำนี้อาจรั่วไหลข้อมูลระบุตัวตนส่วนบุคคล (PII) ละเมิดความยินยอม หรือถูกย้อนกลับเพื่อสร้างข้อความเดิมได้ วัตถุประสงค์ของกลุ่มการควบคุมนี้คือการเสริมความแข็งแกร่งของกระบวนการจัดการความทรงจำและฐานข้อมูลเวกเตอร์ เพื่อให้การเข้าถึงเป็นไปตามสิทธิ์น้อยที่สุด การฝังข้อมูลคำนึงถึงความเป็นส่วนตัว เวกเตอร์ที่จัดเก็บจะมีวันหมดอายุหรือสามารถเพิกถอนตามคำขอ และความทรงจำของแต่ละผู้ใช้จะไม่มีทางปนเปื้อนกับข้อความแจ้งหรือผลลัพธ์ของผู้ใช้อื่น

| 8.1.2 | ตรวจสอบให้แน่ใจว่า API keys หรือ JWTs มีการระบุสิทธิ์แบบกำหนดขอบเขต (เช่น รหัสคอลเลกชัน, คำกริยาแอ็กชัน) และมีการเปลี่ยนหมุนเวียนอย่างน้อยทุกไตรมาส   |   1   |  D/V  |
| 8.1.3 | ตรวจสอบให้แน่ใจว่าการพยายามยกระดับสิทธิ์ (เช่น การสอบถามความคล้ายคลึงข้ามเนมสเปซ) ถูกตรวจจับและบันทึกเข้าสู่ SIEM ภายใน 5 นาที                        |   2   |  D/V  |
| 8.1.4 | ตรวจสอบให้แน่ใจว่า vector DB บันทึกการตรวจสอบ subject-identifier, operation, vector ID/namespace, similarity threshold, และ result count ไว้ในบันทึก. |   2   |  D/V  |

## C8.2 การทำความสะอาดและตรวจสอบฝังข้อมูล

ตรวจข้อความล่วงหน้าสำหรับข้อมูลส่วนบุคคล ระบุหรือทำให้เป็นนามแฝงก่อนการแปลงเป็นเวกเตอร์ และสามารถเลือกที่จะประมวลผลเอนบิดดิ้งหลังจากนั้นเพื่อลบสัญญาณที่เหลืออยู่ได้

|   #   | คำอธิบาย                                                                                                                                                                                | ระดับ | บทบาท |
| :---: | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---: | :---: |
| 8.2.1 | ตรวจสอบให้แน่ใจว่าข้อมูล PII และข้อมูลที่ถูกควบคุมถูกตรวจจับผ่านตัวจำแนกอัตโนมัติและถูกปกปิด, แทนที่ด้วยโทเค็น, หรือถูกลบก่อนการฝังข้อมูล (pre-embedding).                              |   1   |  D/V  |
| 8.2.2 | ตรวจสอบให้แน่ใจว่า embedding pipelines ปฏิเสธหรือกักกันข้อมูลนำเข้าที่มีโค้ดที่สามารถรันได้หรือวัตถุที่ไม่ได้เป็น UTF-8 ซึ่งอาจทำให้ดัชนีเกิดการปนเปื้อน                                |   1   |   D   |
| 8.2.3 | ตรวจสอบให้แน่ใจว่ามีการใช้การทำให้ปลอดภัยแบบความเป็นส่วนตัวเชิงความแตกต่างท้องถิ่นหรือแบบมาตรฐานกับการฝังประโยคซึ่งระยะห่างต่อโทเค็น PII ที่ทราบใดๆ ต่ำกว่าค่าธรณีที่สามารถกำหนดได้     |   2   |  D/V  |
| 8.2.4 | ตรวจสอบให้แน่ใจว่าประสิทธิภาพการล้างข้อมูล (เช่น การเรียกคืนการลบข้อมูลระบุตัวตนส่วนบุคคล, การเปลี่ยนแปลงเชิงความหมาย) ได้รับการตรวจสอบอย่างน้อยครึ่งปีต่อครั้งเทียบกับชุดข้อมูลมาตรฐาน |   2   |   V   |
| 8.2.5 | ตรวจสอบให้แน่ใจว่าการตั้งค่าการล้างข้อมูลมีการควบคุมเวอร์ชันและการเปลี่ยนแปลงได้รับการตรวจทานโดยเพื่อนร่วมงาน                                                                           |   3   |  D/V  |

## C8.3 การหมดอายุของหน่วยความจำ การเพิกถอน และการลบทิ้ง

กฎหมาย GDPR "สิทธิที่จะถูกลืม" และกฎหมายที่คล้ายกันต้องการการลบข้อมูลอย่างทันท่วงที ดังนั้น ระบบจัดเก็บเวกเตอร์จึงต้องรองรับการตั้งเวลาหมดอายุ (TTL), การลบอย่างถาวร, และการทำเครื่องหมายลบ (tomb-stoning) เพื่อป้องกันไม่ให้เวกเตอร์ที่ถูกเพิกถอนสามารถกู้คืนหรือถูกจัดทำดัชนีใหม่ได้

|   #   | คำอธิบาย                                                                                                                                            | ระดับ | บทบาท |
| :---: | --------------------------------------------------------------------------------------------------------------------------------------------------- | :---: | :---: |
| 8.3.1 | ตรวจสอบให้แน่ใจว่าเวกเตอร์และระเบียนเมตาดาต้าทุกตัวมี TTL หรือป้ายกำกับการเก็บรักษาที่ชัดเจนซึ่งได้รับการเคารพโดยงานล้างข้อมูลอัตโนมัติ             |   1   |  D/V  |
| 8.3.2 | ตรวจสอบให้แน่ใจว่าคำขอลบที่ผู้ใช้เริ่มต้นจะล้างเวกเตอร์ เมทาดาท้า แคชสำเนา และดัชนีย่อยภายใน 30 วัน                                                 |   1   |  D/V  |
| 8.3.3 | ตรวจสอบให้แน่ใจว่าการลบเชิงตรรกะ ตามมาด้วยการทำลายข้อมูลด้วยวิธีการเข้ารหัสของบล็อกที่เก็บข้อมูล หากฮาร์ดแวร์รองรับ หรือโดยการทำลายกุญแจในคลังกุญแจ |   2   |   D   |
| 8.3.4 | ตรวจสอบให้แน่ใจว่าวิคเตอร์ที่หมดอายุจะถูกยกเว้นออกจากผลการค้นหาเพื่อนบ้านที่ใกล้เคียงภายในเวลาไม่เกิน 500 มิลลิวินาทีหลังจากหมดอายุ                 |   3   |  D/V  |

## C8.4 ป้องกันการกลับทิศทางและการรั่วไหลของการฝังข้อมูล

แนวทางป้องกันล่าสุด—การซ้อนทับเสียงรบกวน, เครือข่ายฉายภาพ, ความผิดปกติของนิวรอนส่วนบุคคล, และการเข้ารหัสชั้นแอปพลิเคชัน—สามารถลดอัตราการกลับทวนระดับโทเค็นให้ต่ำกว่า 5% ได้

|   #   | คำอธิบาย                                                                                                                                                                                      | ระดับ | บทบาท |
| :---: | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---: | :---: |
| 8.4.1 | ตรวจสอบให้แน่ชัดว่ามีโมเดลภัยคุกคามอย่างเป็นทางการที่ครอบคลุมการโจมตีแบบ inversion, membership และ attribute-inference และมีการทบทวนทุกปี                                                     |   1   |   V   |
| 8.4.2 | ตรวจสอบให้แน่ใจว่าการเข้ารหัสชั้นแอปพลิเคชันหรือการเข้ารหัสแบบค้นหาได้ปกป้องเวกเตอร์จากการอ่านโดยตรงโดยผู้ดูแลโครงสร้างพื้นฐานหรือพนักงานคลาวด์                                               |   2   |  D/V  |
| 8.4.3 | ตรวจสอบให้แน่ใจว่าพารามิเตอร์การป้องกัน (ε สำหรับ DP, สัญญาณรบกวน σ, อันดับการฉายภาพ k) มีความสมดุลระหว่างความเป็นส่วนตัว ≥ 99 % การปกป้องโทเค็น และประโยชน์ใช้สอย ≤ 3 % การสูญเสียความแม่นยำ |   3   |   V   |
| 8.4.4 | ตรวจสอบให้แน่ใจว่าเมตริกความทนทานต่อการย้อนกลับเป็นส่วนหนึ่งของเกณฑ์การปล่อยสำหรับการอัปเดตโมเดล โดยมีการกำหนดงบประมาณการถดถอยไว้แล้ว                                                         |   3   |  D/V  |

## C8.5 การบังคับใช้ขอบเขตสำหรับหน่วยความจำเฉพาะผู้ใช้

การรั่วไหลข้ามผู้เช่า (cross-tenant leakage) ยังคงเป็นความเสี่ยงสูงสุดของ RAG: การกรองคำค้นหาความคล้ายคลึงที่ไม่เหมาะสมอาจเผยเอกสารส่วนตัวของลูกค้ารายอื่นได้

|   #   | คำอธิบาย                                                                                                                                           | ระดับ | บทบาท |
| :---: | -------------------------------------------------------------------------------------------------------------------------------------------------- | :---: | :---: |
| 8.5.1 | ตรวจสอบให้แน่ใจว่าการค้นคืนข้อมูลทุกรายการได้รับการกรองภายหลังตามรหัสผู้เช่า/ผู้ใช้ก่อนที่จะส่งไปยังคำสั่ง LLM                                     |   1   |  D/V  |
| 8.5.2 | ตรวจสอบให้แน่ใจว่าชื่อคอลเลกชันหรือตัวระบุแบบมีเนมสเปซถูกเติมเกลือ (salted) ต่อผู้ใช้หรือผู้เช่าเพื่อป้องกันไม่ให้เวกเตอร์ชนกันข้ามขอบเขตต่างๆ     |   1   |   D   |
| 8.5.3 | ตรวจสอบให้แน่ใจว่าผลลัพธ์ความคล้ายคลึงที่สูงกว่าค่าธรณีประตูระยะทางที่ตั้งค่าได้แต่เกินขอบเขตของผู้เรียกจะถูกทิ้งและกระตุ้นการแจ้งเตือนความปลอดภัย |   2   |  D/V  |
| 8.5.4 | ตรวจสอบให้แน่ใจว่าการทดสอบความทนทานแบบหลายผู้เช่า จำลองคำถามที่เป็นศัตรูซึ่งพยายามดึงเอกสารที่อยู่นอกขอบเขต และแสดงให้เห็นว่าไม่มีการรั่วไหลใดๆ    |   2   |   V   |
| 8.5.5 | ตรวจสอบให้แน่ใจว่ากุญแจการเข้ารหัสแยกกันตามผู้เช่า เพื่อรับประกันการแยกทางคริปโตกราฟฟี แม้ว่าจะใช้ที่เก็บข้อมูลทางกายภาพร่วมกันก็ตาม               |   3   |  D/V  |

## C8.6 ระบบความปลอดภัยหน่วยความจำขั้นสูง

การควบคุมความปลอดภัยสำหรับสถาปัตยกรรมหน่วยความจำขั้นสูงรวมถึงหน่วยความจำเหตุการณ์ (episodic), หน่วยความจำเชิงความหมาย (semantic), และหน่วยความจำการทำงาน (working memory) โดยมีข้อกำหนดเฉพาะด้านการแยกและการตรวจสอบความถูกต้อง

|   #   | คำอธิบาย                                                                                                                                                                                                                                                                                          | ระดับ | บทบาท |
| :---: | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---: | :---: |
| 8.6.1 | ตรวจสอบให้แน่ใจว่าประเภทความจำต่างๆ (episodic, semantic, working) มีบริบทความปลอดภัยแยกจากกันโดยใช้การควบคุมการเข้าถึงตามบทบาท กุญแจการเข้ารหัสแยกต่างหาก และรูปแบบการเข้าถึงที่มีเอกสารสำหรับแต่ละประเภทความจำ                                                                                   |   1   |  D/V  |
| 8.6.2 | ตรวจสอบให้แน่ใจว่ากระบวนการรวมความทรงจำมีการตรวจสอบความปลอดภัยเพื่อป้องกันการแทรกแซงของความทรงจำที่เป็นอันตรายผ่านการทำความสะอาดเนื้อหา การตรวจสอบแหล่งที่มา และการตรวจสอบความสมบูรณ์ก่อนการจัดเก็บ                                                                                               |   2   |  D/V  |
| 8.6.3 | ตรวจสอบให้แน่ใจว่าคำค้นคืนหน่วยความจำได้รับการตรวจสอบความถูกต้องและทำความสะอาดข้อมูลเพื่อตัดป้องกันการสกัดข้อมูลที่ไม่ได้รับอนุญาตผ่านการวิเคราะห์รูปแบบคำค้น การบังคับใช้การควบคุมการเข้าถึง และการกรองผลลัพธ์                                                                                   |   2   |  D/V  |
| 8.6.4 | ตรวจสอบให้แน่ใจว่า กลไกการลืมข้อมูลในหน่วยความจำ ลบข้อมูลที่ละเอียดอ่อนอย่างปลอดภัยโดยมีการรับประกันการลบด้วยวิธีการเข้ารหัส เช่น การลบกุญแจ การเขียนทับหลายรอบ หรือการลบอย่างปลอดภัยโดยใช้ฮาร์ดแวร์พร้อมใบรับรองการตรวจสอบ                                                                       |   3   |  D/V  |
| 8.6.5 | ตรวจสอบให้มั่นใจว่าความสมบูรณ์ของระบบหน่วยความจำได้รับการตรวจสอบอย่างต่อเนื่องเพื่อตรวจหาการแก้ไขที่ไม่ได้รับอนุญาตหรือความเสียหายผ่านทางการตรวจสอบค่ารวม (checksums) บันทึกการตรวจสอบ (audit logs) และการแจ้งเตือนอัตโนมัติเมื่อเนื้อหาของหน่วยความจำมีการเปลี่ยนแปลงนอกเหนือจากการดำเนินงานปกติ |   3   |  D/V  |
